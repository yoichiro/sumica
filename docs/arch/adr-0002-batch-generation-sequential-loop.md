# ADR 2: バッチ生成をクライアント主導の逐次ループとして実装する

## Context

複数枚の画像をまとめて生成したいという要望があり、実現方法としてStable Diffusion自体が持つBatch Countパラメータを使う案と、クライアント側で1枚ずつ生成をループする案が考えられました。洋一郎さんは「Stable DiffusionのBatch Countは使わず、1枚1枚都度頼んでください」と明確に指定し、既存の単一画像生成フローとの一貫性を優先しました。

## Decision

バッチ生成はサーバーを変更せず、クライアント側で既存の単一画像用`/api/generate`エンドポイントを順番に呼び出すループとして実装します。

- 単一生成とバッチ生成で処理が乖離しないよう、`enhanceOnce` / `generateImage` / `persistResult` / `generateAndPersist`という共有コアを抽出しました。検討した3案のうち、単一画像生成の共有コアを抽出する案（単一生成を「バッチのN=1」として扱う）を採用し、バッチ専用関数を別途用意する案、`handleGenerate`に枚数引数を追加する案は却下しました。これにより両者の挙動が将来乖離するリスクを避けられます。
- 各ジョブは`{ width, height, model? }`の正規化された`BatchJob`として表現し、枚数指定・サイズの組み合わせ（幅×高さの直積、候補は512/768/1024、最大16通りに制限）・モデル切り替えの3モードすべてが同じジョブリストと同じ逐次ループに乗ります。
- Seed値はバッチ専用の特別扱い（自動インクリメントやランダム化）をせず、既存のSeedロック設定にそのまま従う仕様としました。同一プロンプト・同一Seedでもサイズやモデルを変えれば出力は変化する、という指摘を踏まえ、将来のパラメータスイープ機能に拡張できる形は残しつつ、今は実装しないと判断しました（YAGNI）。
- モデル切り替えモードは当初「利用可能な全モデルを自動で巡回し、選択UIなし」という仕様でしたが、実際に使ってみた結果を踏まえて後日、チェックボックスで対象モデルを選べるUI（`Set<string>`で管理し、モーダルを開くたびに全選択にリセット）に変更しました。

## Status

承認済み

## Consequences

- サーバー側は一切変更されておらず、既存の単一画像生成エンドポイントのみでバッチ機能全体が成立しています。
- 各画像ごとに進捗表示・逐次的な永続化・失敗時のcontinue-on-failureが自然に手に入り、バッチ専用のサーバー側ジョブ管理は不要でした。
- 一方で、SDのネイティブBatch Countが持つサーバー側での効率化（同一モデルロード状態の使い回しなど）は活用できず、生成時間はN回分のHTTPリクエストのオーバーヘッドを含みます。
- モデル選択UIの追加は、当初の「自動で全部」という設計が実運用で使いにくいと分かったための後発の方向転換であり、設計時点で全てを見通せていたわけではないことを示す一例です。
