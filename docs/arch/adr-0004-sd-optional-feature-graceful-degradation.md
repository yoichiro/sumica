# ADR 4: SDバージョン差異に対し、オプション機能のプロキシを段階的にデグレードさせる

## Context

AUTOMATIC1111/Forgeにはバージョンによって対応APIエンドポイントの差があります。例えば`/sdapi/v1/schedulers`はAUTOMATIC1111 1.9以降または最近のForgeにしか存在せず、古いビルドでは404になります。ノイズスケジューラー（Karras/Exponentialなど）をサンプラーとは独立した軸として選択できるようにする機能を追加するにあたり、この互換性差異への対処方針を決める必要がありました。

## Decision

新しい`GET /api/sd-schedulers`エンドポイントでSDの`/sdapi/v1/schedulers`をプロキシしますが、404などで失敗した場合はエラーを返さず空配列に縮退させます。クライアントはこの一覧が空の場合、スケジューラーのピッカーUIそのものを非表示にし、txt2imgへのリクエストペイロードからも`scheduler`系フィールドを省略します。これにより、フィールドの存在によって古いSDビルドを壊すことを防ぎます。

一方でサンプラー（`sampler_name`）は必須パラメータとして扱い、取得できない場合はピッカーがエラー状態になる仕様のままとしました。「必須のサンプラー」と「オプションのスケジューラー」を明確に非対称に扱うトレードオフです。

## Status

承認済み

## Consequences

- SDのバージョンに関わらずアプリ全体が動作し続けるようになりましたが、古いSDビルドではスケジューラー選択機能が黙って使えなくなるため、ユーザーからは機能追加に気づきにくいという副作用があります。
- この「プロキシが失敗したら空配列に縮退し、UIごと隠す」というパターンは、後のHires.fix用アップスケーラーピッカー（[[adr-0005-hires-fix-support]]）や生成キャンセル機能（[[adr-0006-generation-cancel-interrupt]]）でも再利用され、SDバージョン互換性を扱う際の標準パターンとして定着しました。
- SD モデルピッカー (`/api/sd-models`) は、この一般パターン (「一覧が空なら UI ごと隠す」) に対して意図的に異なる方針を選んでいます。取得失敗時は select 要素を無効化した状態のまま残し、「モデル取得失敗」等のヒントメッセージを表示します。これは、モデル一覧の欠如が (i) SD 側の一時的な接続断か、(ii) SD が起動していない・API に到達できないなど恒久的な問題かを、ユーザーが区別できるようにするための差別化です。スケジューラーや Hires.fix アップスケーラー等のオプション機能は「失敗しても本体機能は使える」ためピッカーごと隠すのが自然ですが、モデル選択は「そもそも生成に不可欠」であり、失敗を UI 上で隠すとユーザーが状況を誤解する可能性が高いためこの分岐を採用しました。
