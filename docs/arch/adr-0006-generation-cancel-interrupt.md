# ADR 6: 生成キャンセルをサーバー側フラグとSDのinterrupt呼び出しで実現する

## Context

Hires.fixの導入などで1回の生成にかかる時間が伸び、進行中の生成を途中で止めたいという要望が生まれました。実現方式として次の3案を比較しました。

- (A) クライアント側のfetchを`AbortController`で単純に中断する案
- (B) サーバーに中断用エンドポイントを追加し、SDの`/sdapi/v1/interrupt`も呼ぶ案
- (C) ジョブID・キューを持つ本格的なジョブ管理システムを構築する案

(C)はシングルローカルユーザー向けツールには過剰と判断し却下しました。

## Decision

(B)を採用しますが、さらに単純化し、クライアント側の`AbortController`は不要としました。SDの`/sdapi/v1/interrupt`を呼ぶと、進行中の`/sdapi/v1/txt2img`が中途半端な画像を持って即座に応答を返すためです。

- クライアントは軽量な`POST /api/generate/interrupt`を呼び出し、サーバーはモジュールレベルの`cancelRequested`フラグを立てたうえでSDのinterruptを呼びます。
- 中断された`txt2img`の応答が返ってきた時点でハンドラーがこのフラグを確認し、永続化をスキップして`{ success: false, cancelled: true }`を返します。
- このツールではSDが同時に1件しかジョブを処理しないため、単一のモジュールレベルフラグで十分と判断し、ジョブIDによる管理は行いません。この1つのエンドポイント・フラグは単一生成・バッチ生成の両方で共通に使え、バッチはキャンセルされた結果を受け取った時点でループを止めるだけで済みます。
- 同じセッションで、Hires.fixのsecond passが原因でtxt2imgが旧来の180秒タイムアウトを超えることが判明し、タイムアウトを600秒に引き上げました。

## Status

承認済み

## Consequences

- 実装がシンプルになり、既存の「SDプロキシエンドポイントを追加する」パターン（[[adr-0004-sd-optional-feature-graceful-degradation]]）を再利用できました。
- 単一フラグ方式はマルチユーザー化やジョブの並行実行には対応できませんが、現状の「ローカル・シングルユーザー」という前提の範囲では十分です。将来複数ジョブを並行処理する必要が生じた場合は、この設計の前提から見直す必要があります。
