# ADR 9: SDXLモデルの判定を`.safetensors`ヘッダー解析に置き換える

## Context

[[adr-0003-sd-sdxl-model-detection-heuristic]]では、チェックポイント名に`"xl"`という文字列が含まれるかどうかで判定する、低コストな名前ヒューリスティックフィルタを採用していました。この方式は、命名規則に反するファイル名（`"xl"`を含まないSDXLモデル、あるいは`"xl"`を含むSD1.5モデル）が存在した場合に誤判定するという限界を、既知のトレードオフとして受け入れていました。

洋一郎さんから、ファイル名に依存しない、より正確な判定手段に置き換えたいという要望がありました。要件としては、一覧に表示されるモデルの外部挙動は現状（SDXL系を除外する）を維持したまま、内部の判定ロジックだけを置き換えることが求められました。

調査の結果、AUTOMATIC1111/Forgeの`/sdapi/v1/sd-models`が返す`filename`フィールドから、サーバーがチェックポイントファイルの実パスを取得できることが分かりました。この環境ではStable Diffusion Web UIがWindows上でネイティブに動作し、Node.jsサーバーはWSL2上で動作しているため、`filename`はWindows形式の絶対パス（例: `E:\stable-diffusion-webui\models\Stable-diffusion\xxx.safetensors`）で返ってきます。これをWSL2のマウントパス（`/mnt/e/...`）に変換すれば、サーバープロセスから同じファイルに直接アクセスできることを実機で確認しました。

`.safetensors`形式はファイル先頭8バイトがヘッダー長（リトルエンディアンの64bit整数）、続く指定バイト数がテンソルメタデータのJSONという構造になっており、ファイル全体を読み込まずにヘッダー部分だけを読めます。このヘッダーの中身を実際のモデルファイル群（稼働中のSD環境、21モデル）に対して検証したところ、以下が分かりました。

- SDXLのbaseモデルは、テキストエンコーダーを`conditioner.embedders.0`（CLIP-L）と`conditioner.embedders.1`（OpenCLIP-bigG）の2つ持つ。
- SDXLのrefinerモデル（`sd_xl_refiner_1.0.safetensors`）は、テキストエンコーダーが`conditioner.embedders.0`の1つのみで、cross-attentionの次元もbaseの2048ではなく1280。そのため「デュアルエンコーダーの有無」だけを条件にすると、refinerを誤って除外対象から漏らしてしまうことが実データで判明しました。
- SD1.5系モデルは`cond_stage_model.*`というキー命名を使い、`conditioner.embedders.*`というキーは持ちません。
- Flux系アーキテクチャのモデル（`2758FluxAsianUtopian_v60SchnellFp8Noclip.safetensors`）も一覧に混在しており、これは`cond_stage_model.*`にも`conditioner.embedders.*`にも一致しません。洋一郎さんの実運用ではこのFluxモデルは問題なく画像生成に使えており、外部挙動を維持する（除外しない）必要がありました。

以上から、判定ロジックは「SD1.5かどうか」ではなく「SDXLかどうか」を陽性判定する設計にする必要がありました。そうすることで、Flux等の未知アーキテクチャは自動的に「除外しない」側に倒れ、現状の外部挙動を壊しません。

## Decision

`/api/sd-models`のモデル一覧フィルタを、名前ヒューリスティックから`.safetensors`ヘッダー解析に置き換えます。

- 各チェックポイントの`filename`が取得できる場合、ファイル先頭のヘッダー（8バイトの長さプレフィックス + 続くJSON形式のテンソルメタデータ）だけを読み込み、`conditioner.embedders.`から始まるキーが1つでも存在すればSDXL系（baseまたはrefiner）と判定して一覧から除外します。
- `filename`が取得できない、またはファイルの読み込みに失敗した場合は、[[adr-0003-sd-sdxl-model-detection-heuristic]]で採用していた名前ヒューリスティック（`"xl"`を含むか）にフォールバックします。ファイルシステムの一時的な問題で判定不能になっても、モデル一覧そのものが壊れないようにするためです。
- SDがWindows上でネイティブに動作し、Node.jsサーバーがWSL2上で動作する構成に対応するため、`filename`がWindows形式の絶対パス（`X:\...`）である場合に限り、WSL2のマウントパス（`/mnt/x/...`）へ変換するヘルパーを追加します（`process.platform === 'linux'`のときのみ変換し、それ以外の環境ではパスをそのまま使います）。

## Status

承認済み — [[adr-0029-sd-sdxl-architecture-ui-handling]] により「SDXL モデルを UI から除外する」挙動は撤廃されました。`.safetensors` ヘッダー解析のロジックそのものは残り、モデル種別 (`sd15`/`sdxl`) の分類目的で本 ADR の設計を継続利用します。分類結果を「除外」に使うか「トグルによる単一情報源」に使うかの選択は ADR 29 の判断が優先されます。

## Consequences

- 稼働中のSD環境（21モデル）に対する実機検証で、新しい判定ロジックの結果は既存の名前ヒューリスティックの結果と完全に一致しました。外部挙動（一覧に表示されるモデルの集合）は変わっていません。
- ファイル名の命名規則に依存しなくなったため、[[adr-0003-sd-sdxl-model-detection-heuristic]]で許容していた「命名規則に反するファイルの誤判定」という既知の限界が解消されました。
- サーバーとSDのチェックポイントファイルに、サーバープロセスから直接アクセスできることが前提になります。ネットワーク越しに離れたマシンでSDを動かす構成では、`filename`のパスにアクセスできず判定不能になりますが、その場合は名前ヒューリスティックへ自動的にフォールバックするため、モデル一覧の提供自体は継続します。
- `/api/sd-models`の応答生成時に、モデルごとに`.safetensors`ファイルの先頭部分を読み込むI/Oが追加されました。ヘッダー部分のみの読み込みのため負荷は小さく、このエンドポイントの呼び出し頻度もクライアント側でマウント時とSD再接続時のみと低いため、キャッシュ等の追加最適化は行っていません。
- Flux等、SD1.5にもSDXLにも一致しないアーキテクチャのモデルは、判定上「除外しない」側に倒れます。これは意図した挙動です。
