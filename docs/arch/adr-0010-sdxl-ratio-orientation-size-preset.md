# ADR 10: SDXLの解像度指定を「アスペクト比+向き+サイズ」プリセット式に切り替える

## Context

これまでSDXLとSD1.5のどちらでも、解像度は「幅」と「高さ」を独立した`<select>`から選ぶUIになっていました。SDXL向けには9値のリスト（576, 768, 832, 896, 1024, 1152, 1216, 1344, 1728）を用意し、幅×高さの直積で81通りの組み合わせが選べる状態でした。

この設計を実運用してみたところ、以下の問題が明らかになりました。

- **多くの組み合わせが破綻しやすい**：SDXLは~1MPを目標にバケット学習されたモデルで、生成品質は総ピクセル数と学習バケットからの距離に大きく依存します。81通りのうち、
  - 0.6MP未満の小さすぎる組み合わせはディテールが消失（例：832×576=13:9、576×896=9:14など）
  - 1.8MPを超える大きすぎる組み合わせはキャラの複製・構図破綻（例：1728×1024=27:16、1728×1216=27:19など）
  - SDXLの学習バケット比率（1:1, 9:7, 19:13, 7:4, 12:5, 3:1）から遠い比率も構図が壊れやすい

  実際に破綻ゾーンに落ちるのは18通り前後で、UIから「事故りやすい選択肢」を排除すればいいのでは、という案がまず出ました。

- **ユーザー体験としてバケットの概念が見えない**：「16:9の壁紙が欲しい」と考えたときに、1024×576とか1344×768とか、頭で計算する必要がありました。SDXLの学習バケットに一致する組み合わせを「知っている人だけ」が選べる状態で、UIがモデルの内部構造をまったく反映していない状態でした。

洋一郎さんから、破綻マスを個別に除外するのではなく、「比率と大きさ」を直接選ぶUIに切り替えたいという提案がありました。SDXLはそもそもアスペクト比バケットで学習されているため、モデルの設計と1対1で対応するUIになるだろう、という観察でした。

具体的なUI設計は4つの質問で1つずつ合意しました。

1. **比率のポリシー**：SDXL純正比率（1:1, 9:7, 19:13, 7:4, 3:1）だけ出すか、一般名（4:3, 3:2, 16:9, 21:9）も出すか → 両方併記し、SDXL純正には⭐バッジを付ける方針
2. **サイズの粒度**：バケット固定・2段階・3段階の3案 → プレビュー用（S≈0.6MP）/ SDXLネイティブ（M≈1MP ⭐）/ 高精細（L≈1.4MP）の3段階
3. **SD1.5の扱い**：一緒に新UIへ揃えるか、従来のまま残すか → SD1.5はバケット学習ではないので、従来の幅×高さ`<select>`のまま
4. **バッチ生成の「サイズ」モード**：SDXLの新UIに合わせて何軸のクロス積にするか → 比率×向き×サイズの3軸クロス積

## Decision

SDXL選択時のメインフォーム解像度ピッカーと、バッチ生成の「サイズの組合せ」モードを、次のプリセット式に置き換えます。SD1.5は従来のUIを維持します。

- 定数`SDXL_PRESETS`（`client/src/App.tsx`）で7つのアスペクト比を定義：`1:1`, `4:3`, `9:7`, `3:2`, `16:9`, `21:9`, `3:1`。各比率にS/M/Lの3サイズ、非正方形は横向き（landscape）と縦向き（portrait）を持ちます（正方形は向き選択なし）。合計39通り。
- 各`M`サイズは可能な限りSDXLの学習バケットに一致させます：
  - `1:1 M` → 1024×1024（SDXL純正バケット）
  - `4:3 M` → 1152×832（SDXL 18:13バケット、4:3の近似）
  - `9:7 M` → 1152×896（SDXL純正、一般名対応なし）
  - `3:2 M` → 1216×832（SDXL 19:13バケット、3:2の近似）
  - `16:9 M` → 1344×768（SDXL 7:4バケット、16:9の近似）
  - `21:9 M` → 1536×640（SDXL 12:5バケット、21:9の近似）
  - `3:1 M` → 1728×576（SDXL純正、一般名対応なし）

  SDXLが実際に学習した比率と一致するのは`1:1`, `9:7`, `3:1`の3つのみで、`4:3`/`3:2`/`16:9`/`21:9`は人間の直感的な一般名を優先し、内部的にはSDXLの近縁バケットへマップしています。UIでは学習バケット比率にのみ⭐マークを付け、Mサイズのバケット一致にもそれぞれ⭐を表示します。

- 全ピクセル値は原則64の倍数（SDXLの推奨単位）で構成しました。Lサイズ（1.3〜1.5MP狙い）では一部の比率で64の倍数を確保できないため、そこは32の倍数に緩和しています。
- 同じ`(width, height)`を複数の比率が指してしまう状況を避けるため、`9:7 L`は当初案の1344×1024から1408×1088に変更しました（1344×1024は`4:3 L`が既に使っている）。加えてバッチ生成側にも`(width, height)`ベースのデデュープを入れ、将来のプリセット編集で偶発的な重複が発生しても同一寸法を2度生成しないようにしました。
- データモデルは`{ width, height }`のまま不変です。ピッカーが選択されるたびに`useEffect`が`resolveSdxlDimensions()`を通してwidth/heightを再計算する構成で、生成パイプライン、Firebase永続化、履歴表示、メタデータは一切変更していません。アーキテクチャ切替時は`findSdxlSelection(width, height)`で現在の値からピッカーの状態を逆引きし、一致しなければ`1:1 M`（1024×1024）にフォールバックします。
- SDXLのバッチ生成「サイズ」モードは、比率・向き・サイズの3軸をそれぞれ独立にマルチ選択し、クロス積でジョブリストを組みます。1:1は向き選択に関わらず1枚扱いに畳み、[[adr-0002-batch-generation-sequential-loop]]で定義した`MAX_SIZE_COMBINATIONS = 16`のキャップは維持します（デフォルトの全選択状態では39通り > 16なので、生成ボタンは自然にdisabledになり、ユーザーが絞り込む必要があります）。
- SD1.5の解像度UI・バッチのサイズモードは従来通り（`SD15_SIZE_OPTIONS = [512, 768, 1024]`の幅×高さ独立選択）を維持します。SD1.5はバケット学習ではないため、比率抽象化のメリットが薄いと判断しました。
- 新UIの発動条件は「モデル種別トグルがSDXL」であり、これは[[adr-0009-safetensors-header-sdxl-detection]]で確立した`.safetensors`ヘッダー解析によるSDXL判定と直接連動します。

## Status

承認済み

## Consequences

- 選択できる組み合わせが81通り → 39通りに半減し、そのうち大多数が生成品質の観点で「安全ゾーン」に収まります。破綻確率の高かった18通り前後の組み合わせは、そもそもUIから到達できなくなりました。
- ユーザーが「16:9の壁紙」「1:1のアイコン」といった直感的な語彙で解像度を選べるようになりました。SDXLのバケット学習という内部構造を理解していなくても、⭐マークで「モデルが最も得意な選択」が視覚的に分かります。
- SD1.5とSDXLでUIが異なる二重管理になります。ただしSD1.5は現時点で1024²程度までしか実用しないシンプルな用途に留まり、SD1.5用の新UIを設計する労力に見合うメリットは見出せなかったため、意図的に許容しました。将来SD1.5側にもプリセット化の需要が出てくれば追加は容易です。
- `(width, height)`のみを保持するデータモデルは変えていないので、`BatchJob`の形やサーバー側の`/api/generate`は無変更のまま新UIが動作しました。過去のSDXL生成画像は洋一郎さんの手で全削除済みだったため、後方互換の考慮は不要でした。
- SDXL純正の`9:7`, `3:1`など一般名と対応関係のない比率も残しました。実運用してみて存在感が薄ければ後日プルーニングする、と洋一郎さんと合意しています。実装後の自動テスト（Playwrightで7比率×S/M/L×向き＋実生成1枚まで検証）では全ケースが期待通り動作しました。
- SDXL選択時のバッチダイアログは、比率・向き・サイズの3チップ列を持つ設計になり、UIとして縦方向に少し伸びました。1画面に収まる範囲だったのでスクロールは追加していません。
