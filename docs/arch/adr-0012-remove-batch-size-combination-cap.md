# ADR 12: バッチ生成のサイズ組み合わせ上限を撤廃する

## Context

「まとめて生成」の**サイズの組合せ**モードには、[[adr-0002-batch-generation-sequential-loop]]の時点から`MAX_SIZE_COMBINATIONS = 16`という防御的な上限が設定されていました。当時のサイズ候補は幅・高さそれぞれ`[512, 768, 1024]`の3値だけで、直積の理論最大は3×3=9通り。上限16は「候補が増えても暴走しないための余裕」として設定されたキャップでした。

[[adr-0010-sdxl-ratio-orientation-size-preset]]でSDXLの解像度指定を「比率+向き+サイズ」の3軸に切り替えたことで、SDXLのバッチ生成モードでは理論最大が **7比率 × 2向き × 3サイズ = 39通り** に増えました。ADR 10ではこのキャップを「維持する」判断を明示的に書いており、**デフォルトの全選択状態（39通り）では生成ボタンが自動的にdisabledになり、ユーザーが16以下まで絞り込む必要がある**という挙動を意図的に採用していました。

しかし実運用してみると、洋一郎さんから「全パターンを一気に生成して比較したい」という要望が出てきました。特に、新しく試したSDXLモデルの得意な比率を一度に眺めたい・LoRAの効きを比率違いで並べたい、といったユースケースでは、39通りを1クリックで走らせられる方が有用です。キャップ本来の目的（誤操作で意図せず大量ジョブを詰めるのを防ぐ）は、生成ボタンのラベルが「39通り生成する」のように**選択数を数字で明示している**時点で、目視で確認できる状況が担保されています。

つまり「間違って39通り押してしまう」リスクよりも、「意図して39通り走らせようとしてボタンがdisabledになる」不便さの方が現実的に大きい、と判断を変える段階に来ました。

## Decision

`MAX_SIZE_COMBINATIONS`定数と、それに紐づく上限チェックを撤廃します。

- `client/src/App.tsx`から`const MAX_SIZE_COMBINATIONS = 16;`を削除。
- 生成ボタンのdisable判定から`> MAX_SIZE_COMBINATIONS`条件を除去。**空選択のときだけdisable**する形に単純化：
  - SDXL: `sdxlSizeJobs.length === 0`のみ
  - SD1.5: `selectedWidths.length === 0 || selectedHeights.length === 0`のみ
- `CLAUDE.md`のバッチ生成節から`MAX_SIZE_COMBINATIONS = 16`の記述を削除し、「no upper cap」に更新。
- 既存ADRは書き換えない方針（[[adr-0002-batch-generation-sequential-loop]]と[[adr-0010-sdxl-ratio-orientation-size-preset]]は歴史的記録として残す）。本ADRが両者のキャップ関連記述を**部分的に supersede** する形で残す。

生成ボタンのラベルは従来通り「N通り生成する」と選択数を表示するため、ユーザーは押す前に何枚生成されるかを常に視認できます。

## Status

承認済み

- [[adr-0002-batch-generation-sequential-loop]]のDecision節にある「最大16通りに制限」の記述を、本ADRが supersede します。
- [[adr-0010-sdxl-ratio-orientation-size-preset]]のDecision節にある「`MAX_SIZE_COMBINATIONS = 16`のキャップは維持します」の記述、およびConsequences節の「デフォルトの全選択状態では39通り > 16なので、生成ボタンは自然にdisabled」の記述を、本ADRが supersede します。

## Consequences

- SDXLでは全39通り（デフォルト全選択状態）を1クリックで実行できるようになりました。「全パターン比較したい」ユースケースがそのまま走ります。
- SD1.5は元々のサイズ候補が3×3=9通りしかなく、旧キャップの16に到達しなかったため、実質的な挙動変化はありません。
- 誤操作リスクは残りますが、生成ボタンのラベルが「N通り生成する」で数を明示しているので、ユーザーは押す前に選択数を目視できます。バッチ実行中はキャンセルボタン（[[adr-0006-generation-cancel-interrupt]]）で任意タイミングで止められるため、走らせてから気付いても撤退可能です。
- 将来もし「意図せず大量ジョブを詰めた」事故が実際に発生したら、上限復活ではなく、たとえば「N通り以上でconfirmダイアログを挟む」等のガードレールで対処します。UIのdisableは意図的な操作を妨げる副作用が強すぎるという学びが本ADRの動機なので、同じ手段には戻らない方向で。
- キャップの撤廃自体は数行の削除で済むため、実装コスト・レビューコスト・ロールバックコストすべて低いです。
