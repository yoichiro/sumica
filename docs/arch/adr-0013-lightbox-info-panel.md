# ADR 13: ライトボックスの詳細情報を「トグル可能な下部オーバーレイパネル」で提供する

## Context

これまでライトボックス（画像拡大表示）と全画面表示中は、画像そのものしか見えない状態でした。生成に使ったパラメータ（寸法、モデル、Seed、Sampler、Steps、CFG、ハイレス、LoRA、Refiner、VAE）を確認したい場合は、ライトボックスを閉じてプレビュータブのメタデータブロックまたはギャラリータイル横の詳細に戻る必要があり、鑑賞と情報確認が分断されていました。

一方で、ライトボックスの主目的は「画像そのものをじっくり見る」ことであり、情報表示を常時サイドバー化すると画像が圧迫され、鑑賞体験の根本を損なう懸念がありました。

洋一郎さんから、「ライトボックスや全画面表示で画像を表示している際に、その画像の詳細情報を表示するかどうかを選択できるようにしてほしい。表示する場合は、できるだけ画像の邪魔にならず、でも視認性がある程度ある状態で表示してほしい」というリクエストがありました。

設計は 4 つの選択で 1 つずつ合意しました。

1. **レイアウト**：下部オーバーレイ / 右サイドドロワー / 左上コーナーカードの 3 案 → 画像の視認性を最優先し、`object-fit: contain` の縦余白を活用できる **下部オーバーレイ** に決定
2. **トグル操作**：ボタンのみ / ボタン＋キーボードショートカット / 常時表示＋X で閉じるの 3 案 → 既存の Esc / ←→ / Space / F との衝突を避けるため **ボタンのみ**
3. **初期状態**：非表示 / 前回状態を記憶 / 常時表示 の 3 案 → 「画像鑑賞優先」の思想を貫くため **開くたびに非表示リセット**（ただしライトボックス内 ←→ ナビゲーション中は表示状態を維持）
4. **情報スコープ**：フル（プロンプト含む）/ 必須のみ / 必須 + 「もっと見る」の 3 案から、洋一郎さんの再選択で **技術パラメータ 10 項目のみ**（プロンプト類は一切含めない）で確定

## Decision

ライトボックス JSX 内に、詳細情報を表示するための下部オーバーレイパネルと、それを開閉する Info トグルボタンを追加します。実装は `client/src/App.tsx` 内のインライン style と新規 state 1 個のみで完結させ、サーバー・CSS ファイル・Firebase スキーマ・既存の永続化フローには一切手を加えません。

- **表示項目は 10 種類の技術パラメータのみ**：寸法、モデル、Seed、Sampler、Steps、CFG、ハイレス、LoRA、Refiner、VAE。プロンプト類（`originalPrompt` / `enhancedPrompt` / `negativePrompt`）、`scheduler`、`timestamp` / `createdAt` は表示しません。プロンプトはギャラリーの「フォームにロード」機能とプレビュータブのメタデータブロックで既に確認できるため、ライトボックスは「画像を見る場所」に徹します。値のない項目は自動的に行が消える条件付きレンダーで、古い画像でもレイアウトが崩れません。

- **パネルは常時マウント、CSS で見せ隠れ**：Info OFF 時にパネル DOM を unmount する条件レンダー方式では、最初のクリックで DOM 生成→即座に `translateY(0)` になり、トランジションが発火しません。そこで `{lightboxMeta && (...)}` の条件下では `showLightboxInfo` の値に関わらず常にマウントし、`transform: translateY(0) / translateY(100%)`、`opacity: 1 / 0`、`pointer-events: auto / none` の 3 つを `showLightboxInfo` にリンクさせて 0.2s ease でトランジションさせます。`pointer-events: none` により、OFF 状態のパネル領域をクリックしたときに背景クリック（ライトボックスを閉じる）が正常に発火します。

- **Info ボタンは既存ツールバー列の左端に追加**：右端から順に `[閉じる 20px]` `[全画面 72px]` `[→ 124px]` `[← 176px]` `[○選択 228px]` `[★お気に入り 280px]` と並んでいる既存の absolute 配置に対し、**Info を `right: 332px`** で追加します。既存ボタンの `right` 値は 1 つも変えないため、既存レイアウトのリグレッションを起こさない配置です。ON 時は背景を `rgba(255, 255, 255, 0.28)` + 白リングシャドウで、OFF 時は他ボタンと同じ `rgba(255, 255, 255, 0.15)` で表現します。

- **メタデータソースはギャラリー画像と `__preview__` 画像を統一**：`lightboxMeta = lightboxIndex >= 0 ? displayedHistory[lightboxIndex] : (morphSourceKey === '__preview__' ? currentGeneration : null)` の派生値で、ギャラリー履歴からの表示と、プレビュータブの `currentGeneration`（`GenerationData` 型）からの表示を同じコードパスで扱います。`GenerationData` と `GenerationRecord` は、パネルが参照する 10 フィールドすべてについて構造的に互換なので、union 型のまま安全にアクセスできます。

- **開閉状態のリセットタイミング**：`closeLightbox()` の先頭で `setShowLightboxInfo(false)` を呼び、次にライトボックスを開いたときは必ず非表示スタートに戻します。ライトボックス内の ←→ ナビゲーションでは `lightboxUrl` のみ変わり `showLightboxInfo` は触らないため、複数枚のパラメータを連続して比較したいユースケースには対応します。

- **OS 全画面と互換**：パネルは `lightboxRef` の子要素として配置しており、`requestFullscreen()` の対象内に自然に含まれます。特別なコードなしで全画面時も同じ挙動になります。

- **キーボードショートカットは意図的に追加しない**：既存の Esc / ←→ / Space / F と衝突しない `I` キーは候補になり得ますが、「明示的な UI 操作」のシンプルさを優先し、ボタン 1 つに絞ります。将来必要になれば `handleKeyDown` に 1 分岐追加するだけで実装可能なので、YAGNI の判断です。

承認済み

## Consequences

- ライトボックス内で「これは何 Seed で作った？」「何 Sampler？」といった疑問に、ライトボックスを閉じずに答えられるようになりました。特に ←→ で複数枚を比較しながら Info を開きっぱなしにできるため、A/B テスト的な閲覧体験が向上しました。

- 画像鑑賞体験は基本的に阻害されません。ライトボックスを開いた直後は Info OFF なので、以前と同じ「画像だけが見える」状態から始まります。詳細を見たい人だけが 1 クリックで開き、閉じたい人はまた 1 クリックで戻せます。

- パネルは 40vh + `overflowY: auto` で高さがキャップされているため、LoRA 複数 + Hires + Refiner + VAE 全部盛りの画像でも画像領域を極端に潰さず、パネル内スクロールで全項目に到達できます。狭幅ウィンドウでは `flex-wrap: wrap` により行が折り返されます。

- プレビュータブ経由のライトボックスでは、`lightboxIndex < 0` のため既存の ○選択（228px）・★お気に入り（280px）が非レンダーになっているため、Info ボタン（332px）と前の矢印（176px）の間に約 156px の視覚的な穴が空きます。この配置は「ボタン位置を状況で動かさない」という一貫性を優先した結果で、機能上の問題はありません。もし見た目が気になる場合は、`lightboxIndex` の状態に応じて Info の `right` 値を切り替える追加コミットで対応可能です。

- 既存アーキテクチャへの影響はゼロ：`GenerationData` / `GenerationRecord` の型定義は変更なし、`/api/generate` などのサーバー契約は変更なし、Firestore のドキュメントスキーマは変更なし、Firebase Storage のパス構造は変更なし、`storage.rules` は変更なし。純粋に `client/src/App.tsx` へのインライン UI 追加（+106 行）に留まります。

- パネルの表示項目からプロンプトを除外したため、「元プロンプトを再確認したい」場合は引き続き `[[adr-0007-gallery-thumbnail-strategy]]` で扱っているギャラリータイルからの「フォームにロード」経由か、プレビュータブ側のメタデータブロックを見る必要があります。この二重系は「ライトボックス＝画像鑑賞、プレビュータブ＆ギャラリー＝プロンプト詳細」という役割分担として意図的に維持しています。

- パネル内のスクロール位置は React が同じ DOM ノードを再利用する関係で ←→ ナビゲーション時に保持されます。10 項目 + 40vh + 折り返しレイアウトの組み合わせで通常は縦スクロールが発生しないため、実害は極めて限定的と判断しました。将来スクロールを毎回リセットしたくなれば、`useEffect` で `lightboxIndex` を依存に置いて `scrollTop = 0` する 3 行で対応可能です。
