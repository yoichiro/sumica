# ADR 17: バッチ生成のキャンセル取りこぼしを防ぐためクライアント側にもキャンセルフラグを持つ

## Context

[[adr-0006-generation-cancel-interrupt]] で採用したキャンセル機構は、次のような単一の共有フラグに依存しています。

- ユーザーが「生成を止める」をクリック → クライアントが `POST /api/generate/interrupt` を送る
- サーバーがモジュールレベルの `cancelRequested = true` を立て、SD の `/sdapi/v1/interrupt` を呼ぶ
- 中断された `/api/generate` のハンドラが、SD からの応答後にこのフラグを確認して `{ success: false, cancelled: true }` を返す
- クライアントは `GenerationCancelledError` を投げ、[[adr-0002-batch-generation-sequential-loop]] のバッチループがこれを catch して break

この機構は単一生成では問題なく動きますが、バッチ生成では次のシナリオでキャンセルが取りこぼされることが実運用で判明しました。

1. ジョブ N の SD 生成が完了 → クライアントは Firebase Storage に画像とサムネイルをアップロードし、Firestore にメタデータを書き込みます（数秒かかります）
2. その間 SD はアイドル状態です。ユーザーが「生成を止める」をクリックすると、サーバーは `cancelRequested = true` を立てて SD の `/interrupt` を呼びますが、SD にはキャンセルすべきジョブがありません
3. Firebase 永続化が完了し、クライアントがジョブ N+1 の `/api/generate` を送信します
4. **サーバー `/api/generate` ハンドラの先頭に置いた "defensive reset"（`cancelRequested = false`）が、まだ拾われていないキャンセルシグナルを上書きして消してしまいます**
5. ジョブ N+1 は普通に走り、バッチは止まりません

この defensive reset は、単一生成において「生成完了直後にキャンセルが届く」ようなレース条件で `cancelRequested` フラグが true のまま残り、次回の生成が誤ってキャンセルされる副作用を防ぐために意図的に置かれています。単一生成の正しさを保つため削除できません。

サーバー側だけで「単一生成の後始末」と「バッチ生成のジョブ間キャンセル」の両方を満たすには、バッチセッション ID やリクエスト種別による分岐など、状態管理を複雑化させる必要があります。しかしバッチ生成は [[adr-0002-batch-generation-sequential-loop]] の通り**クライアント側の逐次ループ**であり、バッチの開始・終了・現在位置をすでにクライアントが握っています。ここに小さなキャンセルフラグを追加するほうが、サーバーの状態を増やすより自然です。

## Decision

**クライアント側に `useRef` によるバッチ用キャンセルフラグ `batchCancelledRef` を追加し、バッチループの各イテレーションの先頭でチェックすることにします。**

具体的な設計は次の通りです。

- `requestCancel` は従来通り `POST /api/generate/interrupt` を呼び出しますが、その直前に `batchCancelledRef.current = true` を設定します。
- `handleBatchGenerate` はバッチ開始時に `batchCancelledRef.current = false` にリセットし、`for` ループの各イテレーション先頭で `batchCancelledRef.current` を確認します。true であれば、その時点で `cancelledInLoop = true` として `break` し、以降のジョブは送信しません。
- サーバー側の `cancelRequested` フラグと defensive reset はそのまま残します。SD 生成中のキャンセル（インフライトなジョブに対する即時停止）は既存の経路が引き続き処理し、Firebase 永続化中や次ジョブ発射前のギャップに届いたキャンセルは新しいクライアント側フラグが確実に拾います。

代替案として次の 2 案を検討し、いずれも却下しました。

- **案 X: サーバー側の defensive reset を削除する** — 単一生成において「生成完了直後にキャンセルが届く」ケースでフラグが立ちっぱなしになり、次回の生成リクエストが誤ってキャンセルされる副作用があるため却下しました。単一生成の正しさをバッチの都合で崩すべきではありません。
- **案 Y: サーバー側でリクエスト種別（単一 vs バッチ）を判別し、バッチ中は defensive reset を無効化する** — サーバーがバッチの開始・終了・現在ジョブを追跡する必要があり、`clientPersist` と同様のリクエストパラメータを増やすことになります。バッチセッション全体のライフサイクルをサーバー側で管理する状態が増え、[[adr-0002-batch-generation-sequential-loop]] の「バッチはクライアント側の逐次ループにとどめ、サーバーは単一生成 API を提供するだけ」という設計原則に反するため却下しました。

## Status

承認済み

## Consequences

- **バッチ生成中の「生成を止める」が Firebase 永続化フェーズや次ジョブ発射前のギャップでも確実に効くようになりました**。ユーザーが「押しても止まらない」と感じていた失敗モードが解消されます。
- キャンセルフラグが**クライアント側とサーバー側の 2 か所**に存在することになりました。役割分担は「サーバー側フラグ = インフライトな SD ジョブを即時に止めるためのシグナル」「クライアント側フラグ = バッチループ全体を止めるための持続シグナル」と明確に切り分けられ、両者が同時に true になっても矛盾なく動きます。競合ではなく重ね掛けです。
- クライアント側フラグは `useRef` を使うため React の再レンダーを起こさず、実装コストは 4 行程度の追加で済んでいます。バッチ以外のフローには影響を与えません。
- Firebase 永続化中にキャンセルされた場合、「現在ジョブ N の保存は完了する／次のジョブ N+1 以降は送信されない」という挙動になります。ユーザーからはトースト `N 枚生成した時点で止めました🛑` として見えます。この挙動を「もっと即時に止めたい（Firebase アップロードも中断したい）」と感じる場面が出た場合は、`AbortController` を Firebase SDK 経由で伝播させるさらなる拡張が必要になりますが、現時点ではその要求はなく、シンプルさを優先します。
- 将来 [[adr-0006-generation-cancel-interrupt]] の Consequences で言及されている「マルチユーザー化やジョブの並行実行」に対応する場合、サーバー側フラグをジョブ ID ベースに拡張することになります。そのときはクライアント側フラグも「バッチセッション ID + ジョブ ID」に合わせて再設計する必要がありますが、現状の「ローカル・シングルユーザー」前提では、この 2 フラグ方式で必要十分です。
