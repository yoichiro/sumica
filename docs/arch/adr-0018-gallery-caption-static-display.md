# ADR 18: ギャラリータイルのキャプションを静的 2 行レイアウトで実装する

## Context

ギャラリーの各サムネイル画像の下には、その画像を識別するための情報表示エリアがあります。もともとは「生成プロンプトの先頭 30〜40 字を `text-overflow: ellipsis` で切り詰めたもの + 作成日付」の 2 行構成でしたが、日本語自然言語プロンプトの冒頭数十字だけでは画像同士の判別性が低く、実質的に機能していない状態でした。

このエリアで表示したい候補情報は多数あります：モデル名、寸法（W×H + アスペクト比）、作成日時、Sampler + Scheduler、Hires.fix の有無・詳細、適用 LoRA の名前と weight、Seed 値、Refiner、VAE。ただし、サムネイル画像自体のサイズを変えずにキャプション領域だけを工夫する制約下では、実効的な横幅は 1 行あたり 20〜30 文字程度しかありません。この狭さでは、全情報を静的に同時並置することが物理的に不可能です。

同日中に、まず「情報を時間で見せる」発想の**縦スクロール式ローテーション**（[[2026-07-07-gallery-caption-rotation-design]]）を実装しました：3〜6 秒ごとに 2 スロットずつ縦にスクロールして情報を切り替え、hover で個別に停止、`prefers-reduced-motion` に対応、といった凝った作りでした。しかし洋一郎さんが実際に使ってみたところ、以下の 3 つの観察が得られました。

1. **モデル名とサイズは常に見えていたい**。ローテーションで時々見えなくなるとギャラリーをスキャンしづらい。
2. **Hires.fix や LoRA は「適用されているかどうか」だけ分かれば十分**。具体的な数値、LoRA 名、weight などは不要。
3. **日付はあると便利だが、他の情報より視認性が下がっても構わない**。

これは「情報を時間で見せる」設計思想の反対で、「情報を視覚的階層で見せる」アプローチが適していることを意味しました。

## Decision

**ローテーション式を廃止し、静的 2 行レイアウトの `CaptionInfo` コンポーネントで置き換えます。** 実装は [[2026-07-07-gallery-caption-static-design]] に詳述してあります。

具体的な配置は次の通りです。

- **Row 1**: モデル名（太字、`--text-primary` 色、`fontSize: 13px`、はみ出しは ellipsis で省略、`flex: 1`）と、右端に短縮日付（`MM-DD` 形式、`fontSize: 10px`、`--text-muted` 色、`flex-shrink: 0`）を横並びに配置します。
- **Row 2**: 左端にサイズ文字列（例 `1024×1536 (3:2)`、`fontSize: 11px` 中太、`--text-primary` 色、必要なら ellipsis）、右端に絵文字バッジ群（⚡ Hires 適用時、🎭 LoRA 適用時、`fontSize: 12px`、それぞれ `title` 属性でツールチップを付与）を `justifyContent: space-between` で配置します。バッジ 2 つが同時表示のときは、絵文字グリフに含まれる横方向余白による見た目のギャップを詰めるため、LoRA バッジに `marginLeft: '-4px'` の負マージンを当てます。

代替案として次を比較検討し、いずれも却下しました。

- **ローテーション式**（[[2026-07-07-gallery-caption-rotation-design]]）: 実装して実運用したが、上記フィードバックにより「情報を追いづらい」と判断され不採用。
- **静的パラレル（複数チップ同時表示）**: 幅が足りずスマホでも詰まりすぎて実用にならない。狭幅制約が支配的。
- **表示フィールドをユーザーが選ぶ切替 UI**: ローテーションの「情報が流れてくる」体験は損なうが、実装可能。ただし常時見せる情報を絞る本要件の主旨と合わないため不採用。
- **横スクロール（マーキー）で 1 行に詰め込む**: 情報密度は最大だが動きがうるさく、hover 停止と噛み合わない。
- **hover でポップアップ表示**: グリッドを静かに保てるが、touch デバイスで機能しない。
- **lucide-react アイコン（`Zap`, `Palette`）でバッジを表現**: 一貫性は良いが、絵文字（⚡🎭）のほうがコンパクトかつ既に色分けされていて視認性が高い。将来アイコンに寄せたければ差し替えは容易。

## Status

承認済み

## Consequences

- **モデル名とサイズが常時見える**ようになり、ギャラリーを目でスキャンして特定の画像を探す用途で判別性が大きく向上しました。ローテーション式で「今その情報が表示されていない」ストレスは解消しました。
- **Hires.fix と LoRA は「有無」のみを 12px の絵文字バッジで表現**することで、狭い横幅でも 2 つ同時に置ける情報密度を確保しました。具体的な数値や名前が知りたいときはライトボックスの詳細情報パネル（[[adr-0013-lightbox-info-panel]]）で確認する運用に自然に整理されます。
- **`buildCaptionInfo` は純粋関数、`CaptionInfo` は React hooks を使わない小さな関数コンポーネント**として実装できました。以前の rotation 実装が持っていた `useState`（tick、displayTick、scrolling、isHovered）、`useEffect`（`setInterval`、hover ガード）、CSS の `@media (prefers-reduced-motion: reduce)` などのインフラは全て不要になり、コードが大幅にシンプルになりました。
- **Seed、Sampler、Scheduler、Refiner、VAE、Steps、CFG などの詳細技術パラメータはキャプションから完全に外しました**。これらは [[adr-0013-lightbox-info-panel]] のライトボックス下部オーバーレイパネルで見られるため、キャプションは「画像のスキャンに必要最小限」の役割に徹します。
- **反復開発の副産物**として、rotation 実装（Task 1〜5 で構築し、レビューまで通した）は 5 個のコミット（`c199e30..583e4ea`）として git 履歴に残っています。同日中に static 版へ全面書き換えするコミット（`fc31921`）でそれらの実装コードは全て削除されましたが、design spec は [[2026-07-07-gallery-caption-rotation-design]] を Superseded ノート付きで保存してあり、「なぜローテーションが試されて捨てられたか」の記録として参照可能です。**動くコードを作ってから live で判断する**というプロトタイプ駆動のアプローチが、狭い探索領域の UX を確定させる上で有効だったと言えます。
- 実装後、いくつかの微調整コミット（`f3188bf` 日付を Row 2 から Row 1 へ移動、`f3de373`/`a773bf4` バッジ間マージンの詰め、`b5b3eae` サイズ左寄せ・バッジ右寄せ、`c12a501` 左端タイルの hover 時 border クリップ修正）が積まれています。これらは静的レイアウトを崩さず、局所的なピクセル単位の調整で完結しました。特に `c12a501` は「`.scale-hover:hover { transform: scale(1.02) }` によるタイルの拡張分が、スクロール親コンテナの `overflowY: auto` に起因する暗黙的な `overflow-x` クリップに引っかかる」というブラウザ仕様のクセを、`paddingLeft: '4px'` の対称化 1 行で解決しています。
- ダークモード対応やアクセシビリティは自動的に整合します。`--text-primary` / `--text-muted` は [[adr-0008-dark-mode-os-follow-only]] のカラーバリアブルで、絵文字バッジには `title` 属性でスクリーンリーダー向けの意味情報が付与されます。アニメーションゼロなので `prefers-reduced-motion` への配慮も不要になりました。
