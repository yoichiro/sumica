# ADR 19: 履歴ギャラリーの複数選択に Shift+クリックの範囲選択（Gmail 型・加算）を採用する

## Context

履歴ギャラリーの各サムネイルには、右下に選択チェックボックス（`SelectButton`）が重ねられていて、ユーザーはそれをクリックすることで画像を「複数選択」できます。選択した画像はツールバーの一括削除操作で削除できます（[[adr-0007-gallery-thumbnail-strategy]] の永続化フロー参照）。

これまでの実装では、複数枚を選択するにはチェックボックスを 1 個ずつクリックするしかありませんでした。SDXL の実験を回して数十枚〜100 枚以上をまとめて削除したい場面や、日付フィルタで絞ってから「この日の失敗作を一気に消したい」といった作業では、1 枚ずつクリックする手間が明らかに UX の摩擦になっていました。

洋一郎さんから、Shift キーを押しながらクリックすることで一定の範囲を選択できる**一般的な範囲選択機能**を追加してほしいというリクエストがありました。

「一般的な範囲選択」といっても、細部で振る舞いが 2 系統に分かれます。

- **Gmail 型（加算）**: アンカーからターゲットまでの範囲内すべてのタイルを選択状態にする。範囲外の既存選択は保持される。
- **トグル型**: アンカーからターゲットまでの範囲内のタイルの選択状態を反転する（選択済みなら解除、未選択なら選択）。

前者は「範囲内を確実に選択する」明確な意図と噛み合いますが、既に範囲内に含まれるタイルが選択済みの場合は「変化なし」で済むため、ユーザーは常に「範囲内は全部選ばれる」というシンプルな期待でクリックできます。後者は柔軟ですが、範囲内の一部が選択済みで一部が未選択のとき、どちらに揃うのか判断コストが上がります。Gmail・Google Drive・Slack など多くの Web UI が Gmail 型を採用しています。

## Decision

**Gmail 型（加算）の Shift+クリック範囲選択を採用します。** 具体的な設計は次の通りです。

- **アンカー追跡**: 直近にチェックボックスがクリックされたタイルの ID を `useRef` で保持します。選択・解除の区別は問いません（後者からでも Shift+クリックで範囲選択の起点になれる）。ref を使うのは、この値の変化で再レンダーを起こす必要がないためです。
- **通常クリック（Shift なし）**: 既存の `onToggleSelected(id)` を呼び、アンカー ref を今回の ID に更新します。
- **Shift + クリック**: アンカー ref が存在しかつ現在の ID と異なる場合、`displayedHistory` の配列順（＝グリッドの視覚順）でアンカーからターゲットまでのインデックス範囲を計算し、その範囲内のすべての ID を既存の選択セットに追加した新しい `Set` を `onSetSelectedIds` に渡します。範囲計算は方向に依存せず、アンカーがターゲットより後ろにあっても正しく動きます。アンカーまたはターゲットが `displayedHistory` に見当たらない場合（フィルタ変更等の副作用）は、範囲選択を諦めて単一トグルにフォールバックします。
- **ピュア関数として切り出し**: 範囲計算のロジックは `client/src/components/rangeSelection.ts` の `computeRangeSelectionAdd(anchorId, targetId, displayIds, currentSelection): Set<string> | null` として抜き出し、Vitest で 9 個のテスト（forward/backward、既存選択保持、アンカー同一、アンカー未存在、Set 非破壊性など）を書いています。UI 側の React コードには直接テストを書かず、判断ロジックの正しさはこのピュア関数レベルで保証します（[[adr-0015-ui-component-split-with-hybrid-state]] のテスト方針を継承）。
- **Ctrl/Cmd + クリック**: 今回は対応しません。既存の「単一クリックでトグル」の仕組みが実質的に Ctrl/Cmd+クリックの役割を果たすためです（チェックボックス UI ではクリックがそもそもトグル動作）。将来必要になれば `e.ctrlKey || e.metaKey` の分岐を加えて拡張可能な設計にしてあります。

代替案として次を比較検討し、いずれも却下しました。

- **トグル型範囲選択**: 直感的な状況もありますが、範囲内の状態が不均一なときの挙動がユーザーに読みにくくなります。今回は「範囲内を全部選ぶ」という明確な意図のほうが日常操作に合うため不採用。
- **範囲選択で既存選択をクリアしてから範囲だけ選ぶ**（Windows Explorer 型）: 既存選択をうっかり消すリスクがあり、チェックボックス UI の「加算的」なメンタルモデルと合わないため不採用。
- **Shift 押下でドラッグ選択（範囲を矩形で囲む）**: 実装コストが高く、狭いグリッドではオーバーキル。既存の「明示的なチェックボックス」の UI 言語も崩れるため不採用。

## Status

承認済み

## Consequences

- **大量選択が数クリックで完了する**ようになりました。日付フィルタで絞った 50 枚を全選択したい場合、1 枚目をクリック → 最後をシフトクリック、の 2 動作で済みます。
- 既存の選択操作（単一クリックでのトグル、お気に入りボタン、画像クリックによるライトボックス開き）は**一切影響を受けません**。`handleSelectClick` は `e.stopPropagation()` を維持しつつ Shift の有無で分岐するだけで、他の操作パスは共通コードを通りません。
- アンカー ID は `useRef` に持つため、日付フィルタを切り替えて `displayedHistory` の中身が変わったあとで Shift+クリックすると、アンカーが現在の表示範囲に存在せず範囲選択がフォールバックで単一トグルに落ちます。この挙動は `computeRangeSelectionAdd` が `null` を返すことでハンドラ側が検知し、既存の `onToggleSelected` を呼ぶ形で自然に守られます。ユーザーからは「変な範囲が選ばれる」ではなく「単に 1 個選ばれた」と見えるので、混乱は起きにくい設計です。
- 選択状態のリセット（すべて解除ボタン）を押しても、アンカー ref はリセットされません。これは意図的で、リセット直後に「さっき見ていた辺りから」範囲選択を続けたいユースケースを想定しています。もしユーザーの期待とズレることが判明すれば、リセット時に `lastClickedIdRef.current = null` を挟む 1 行で調整可能です。
- ピュア関数として抜き出したことで、[[adr-0015-ui-component-split-with-hybrid-state]] の「テスト可能な純粋ロジックは切り出す」方針にきれいに乗りました。9 個の Vitest ケース（forward/backward/既存選択保持/アンカー同一/アンカー未存在/Set 非破壊性）が範囲計算のリグレッションを継続的にガードします。
- Ctrl/Cmd+クリックには対応しませんが、`handleSelectClick` の分岐に 1 条件加えれば拡張可能な形になっています。実際に必要と感じられた時点で YAGNI を破ればよく、それまでは追加しません。
