# ADR 22: ISC から AGPL-3.0-or-later にリライセンスする

## Context

Sumica はもともと `npm init` が付与した ISC ライセンス（`package.json` の `"license"` フィールドのデフォルト値）のまま公開準備を進めていました。ISC は MIT と同等の許諾型ライセンスで、著作権表示さえ残せば商用利用・改変・再配布・私的利用のすべてが自由という緩い条件です。

OSS として公開する準備を進めるなかで、洋一郎さんから 2 つの懸念が提示されました。

1. **法的責任の懸念**: 本ソフトウェアはコンテンツフィルタや Safe-for-Work 制約を意図的に持たず、Stable Diffusion API（AUTOMATIC1111 / Forge）に対して任意のプロンプトを渡せます。第三者が本ソフトウェアを利用して商用サービスを立ち上げ、そこで「いかがわしい画像の生成」を可能にしてしまった場合、本ソフトウェアの著作者に何らかの罪が問われる可能性があるのか、それを避けたい。
2. **成果の還元性の懸念**: この開発時間で得た成果が独り占めされ、改変版が閉じて商用サービス化されるのは避けたい。誰かがフォークして機能追加した場合、その改変も OSS として還元してほしい。

1 番目の懸念について、主要 OSS ライセンス（MIT / Apache 2.0 / GPL / AGPL 等）はいずれも "NO WARRANTY" / "LIMITATION OF LIABILITY" 条項を持ち、著作者への損害賠償請求を制限する法的保護は各ライセンスに標準で組み込まれています。加えて、日本の刑法上の幇助罪は「具体的な犯罪の認識」と「幇助の意図」を要件とするため、OSS として無差別に公開する行為が単独で幇助罪を構成する可能性は極めて低いという専門家の一般的な見解があります。したがって 1 番目の懸念は、どのライセンスを選んでも `LICENSE` ファイル + README の Disclaimer セクションによって実務上十分に軽減できます。

2 番目の懸念については、ライセンスの選択が直接的な効果を持ちます。MIT / Apache 2.0 のような許諾型は「派生物をクローズドソースの商用サービスとして提供する」ことを許容します（大手クラウドが Elasticsearch / MongoDB / Redis を wrap して SaaS 化した歴史がこれです）。一方 AGPL v3 の Section 13 は「改変版をネットワーク経由で第三者に提供する場合、改変ソースの開示を義務付ける」条項を持ち、この "SaaS ループホール" を塞ぎます。

Sumica が API 経由で依存している AUTOMATIC1111 Stable Diffusion Web UI が **AGPL v3** で公開されている事実も判断材料になりました。「上流と同じ思想で下流も運用する」ことで、コミュニティエコシステム全体の還元性が保たれます。ただし、Sumica は HTTP API 経由で AUTOMATIC1111 を消費しているだけで、AGPL のコピーレフトが自動的に Sumica に感染するわけではありません（FSF の公式見解: ネットワーク越しの疎通は "derivative work" を作らない）。Sumica のライセンス選択は Sumica 側の独立した判断です。

洋一郎さんは、A（誰かが Sumica を fork して閉じた有料サービス化する未来）と B（企業が "AGPL は使えない" ポリシーで採用を諦める未来）を比較して、A のほうが悔しいと明確に判断しました。この価値観に AGPL v3 が最も適合します。

## Decision

**Sumica を AGPL-3.0-or-later にリライセンスします。**

具体的な変更は次の通りです。

- **`LICENSE` ファイル**: GNU が公開する AGPL v3 公式全文（`https://www.gnu.org/licenses/agpl-3.0.txt`、661 行）をリポジトリルートに配置します。
- **`package.json` の `license` フィールド**: ルート・`client/`・`server/` の 3 つで `"AGPL-3.0-or-later"`（SPDX 識別子）に統一します。`-or-later` にすることで、将来 GNU が AGPL v4 等を発行したときにも運用継続できます。AUTOMATIC1111 も同じ `-or-later` 形式です。
- **README の Disclaimer セクション**: 「コンテンツフィルタは意図的に持たない」「生成物の合法性は利用者の責任」「著作者は第三者運用について責任を負わない」「上流ライセンス（AUTOMATIC1111 の AGPL、LM Studio の利用規約）は利用者が独立して遵守する責任がある」を明記します。日本語版と英語版の両方に同一趣旨を掲載します。
- **README の Contributing セクション**: 「PR / issue コメント / パッチを提出することで、貢献内容が AGPL-3.0-or-later ライセンスで提供されることに同意したとみなす」旨を明示します。これにより将来のライセンス問題を回避します。
- **ソースコード先頭のライセンスヘッダは追加しない**: GNU 推奨ではありますが、React コンポーネントを 15 ファイル以上抱える現状で全ファイルにヘッダを維持するのは保守コストが高すぎます。`LICENSE` ファイル + README + `package.json` の 3 点セットで法的には十分機能します。多くの現代 OSS プロジェクト（AUTOMATIC1111 を含む）が同じ判断をしています。

代替案として次を比較検討し、いずれも却下しました。

- **MIT ライセンス**: React / Vite / TypeScript / lucide-react / Firebase SDK など Sumica の依存ライブラリが軒並み MIT / Apache 2.0 系で、ライセンス整合性という意味では最も自然です。しかし派生物のクローズド SaaS 化を許容してしまう点が、洋一郎さんの 2 番目の懸念（成果が独り占めされる）と真っ向から衝突するため却下しました。
- **Apache 2.0**: MIT に特許明示条項を加えた形。個人ツール規模で特許 grant が意味を持つ場面は薄く、SaaS ループホールの問題は MIT と同じく残るため、AGPL の相互扶助原則を優先しました。
- **GPL v3（AGPL でない GPL）**: 派生物のコピーレフトを要求しますが、「配布時のみ」効くため SaaS ループホールが残ります。Web アプリの性質上、AGPL の "ネットワーク経由でも copyleft 発動" 条項が本質的に必要でした。
- **ISC のまま**: `npm init` のデフォルトを流用したもので、意図した選択ではありませんでした。積極的に採用する理由がありません。

なお、ライセンス変更の一般則として「緩い → 厳しい方向は将来変更可能、逆はコントリビュータ全員の同意なしには事実上不可能」という非対称性があります。洋一郎さんが単独著作権者である現時点でこの決定を確定させることで、将来コントリビュータが増えても AGPL のコピーレフトを維持しやすくなります。

## Status

承認済み

## Consequences

- **改変版を SaaS として運用する第三者は、改変ソースの開示義務を負います**。「Sumica を fork してコンテンツフィルタを付け、その修正版を有料 API サービス化」というシナリオでは、その第三者は改変分を AGPL v3 で公開しなければなりません。これは洋一郎さんの 2 番目の懸念に直接的に応える効果です。
- **「AGPL 禁止」の社内ポリシーを持つ企業からの採用は難しくなります**。多くの大手企業は AGPL の transitivity を嫌って社内ブランケット禁止しています。Sumica の想定ユーザーは個人開発者 / 小規模チームなので、ここは実質的な影響が小さいと判断しました。
- **README に Disclaimer セクションが常設**されたことで、「コンテンツフィルタを意図的に持たない」「利用者の責任範囲」「上流ライセンス遵守義務」が明文化されました。第三者運用の法的責任分離が README ベースで可視化されている状態です。
- **貢献規約（Contributing セクション）に "PR 提出 = AGPL 同意" が明記**されたことで、将来コントリビュータが増えた場合のライセンス問題（"自分の PR は MIT のつもりだった" 等）を予防できます。単独開発者の間はほぼ無関係の条項ですが、将来を見据えた保険としては意味があります。
- **README の言語分離**（[[adr-0020-i18n-jp-en-support]] の派生決定として、英語版を default、日本語版を `README_ja.md` に）と併せて、非日本語話者にもライセンス条件が正しく伝わる状態になっています。
- **AUTOMATIC1111 との思想的整合**が取れました。上流 (AGPL v3) → 下流 (Sumica、AGPL-3.0-or-later) が同じ copyleft を継承する形で、Stable Diffusion クライアント / フロントエンドエコシステム全体の還元性を Sumica が損なわない立場になっています。
- **将来ライセンスを緩めることは事実上不可能**になります。単独著作権者である現時点なら理論上可能ですが、コントリビュータが 1 人でも入ったあとは全員の同意なしには MIT 等に戻せません。この非対称性を踏まえ、「まず AGPL で始めて、必要なら緩める」ではなく「AGPL で行き続ける」前提の設計として受け入れました。
