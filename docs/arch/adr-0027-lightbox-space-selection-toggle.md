# ADR 27: ライトボックスに Space 選択トグルを追加し、ギャラリーの選択 Set を共有する

## Context

履歴ギャラリーには「複数選択して一括削除」フローがあり、選択は `selectedIds: Set<string>` として `App.tsx` の state に保持されます。この Set は「削除」ボタン ([[adr-0019-gallery-shift-click-range-selection]]) の対象決定と、選択件数バッジの表示に使われています。

一方でユーザーの実際の作業フローには、「ライトボックスで拡大しながらパッパッ次々めくって、削除したいものだけマークしていき、閉じた後に一気に削除ボタンを押す」というワークフローがあります。ライトボックス側にも選択のトグル手段がないと、削除候補を見つけるたびにライトボックスを閉じてサムネイルの選択チェックを操作し、また拡大表示に戻る、という往復が発生し、拡大しながら判断する体験が損なわれます。

同時に、ライトボックス内で **直接削除** できるようにするかどうかは慎重に検討する必要がありました。拡大中の 1 枚をその場で「削除」できる UI は、確認モーダルを挟むにせよ、隣の画像を含めた文脈が見えない状態で削除判断を下しがちで、誤削除のリスクが上がります。実際 [[adr-0021-favorite-recipe-rollup-ranking]] 系の作業で「削除は必ずギャラリーに戻ってから一覧確認 + 一括削除」が事故防止として機能していました。

もう 1 つ考慮すべきは、ライトボックスがプレビュータブから開かれた場合（`lightboxIndex === -1`、生成直後の画像のみを表示中）は、そもそも「ギャラリー内の選択」という概念が意味を持たないことです。生成直後の 1 枚はまだ履歴に紐づいていない、あるいは `displayedHistory` の中に無いので、選択トグルの対象がありません。

## Decision

**ライトボックス右上のツールバーに選択トグルボタンを配置し、ギャラリー側の `selectedIds` Set をそのまま共有します。ライトボックス内での直接削除は採用しません。** 具体的な設計は次の通りです。

- **ボタン UI**: `lucide-react` の `Circle` / `CheckCircle2` アイコンで、選択状態に応じて切り替えます。既存のギャラリータイル ([[adr-0018-gallery-caption-static-display]]) のセレクトアイコンと同じデザインで、視覚的一貫性を保ちます。
- **Space キーのバインド**: `resolveLightboxKey` に `Space` → `{ type: 'toggleSelect' }` を追加し、`onToggleSelect(lightboxIndex)` を呼びます。マウス操作を挟まずキーボードだけで選択・次画像・選択・次画像を繰り返せる高速フローを提供します。
- **共有する state**: 新しい選択状態を作らず、既存の `selectedIds` Set とその toggler `toggleSelected(itemKey(displayedHistory[idx]))` をそのまま Lightbox に prop で渡します。閉じた後にギャラリー画面に戻れば、そこで選択状態が既に反映されており、「削除」ボタンで一括削除に直接進めます。
- **`lightboxIndex === -1` のガード**: 選択トグルボタンは `lightboxIndex >= 0` のときだけ描画します。プレビュータブから開かれた生成直後のライトボックスでは、ボタンが物理的に表示されません。Space キーの解決器も同じ条件で `null` を返します。
- **直接削除の非採用**: ライトボックス内には「削除」ボタンを一切置きません。誤削除防止 + フローの一貫性（削除は必ずギャラリー確認モーダル経由）のためです。

代替案として次を比較検討し、いずれも却下しました。

- **ライトボックス内に「削除」ボタンを置く**: 拡大中の 1 枚をその場で削除できる速さは魅力ですが、隣の画像との比較文脈が失われた状態で削除判断を下すことになり、誤削除のリスクが確実に上がります。かつ [[adr-0021-favorite-recipe-rollup-ranking]] の rollup 集計に対する `writeBatch` 減算の複雑さも 1 箇所に集約したく、削除は `handleDeleteSelected` を通す原則で統一しました。
- **選択状態をライトボックス専用に持つ**: 独立した Set を持たせて閉じたらギャラリー側にマージする案もありましたが、複数箇所での二重管理は不整合の温床です。既存 Set を共有するのが最もシンプルで、`displayedHistory` の並びも同じソースなのでインデックスも一貫します。
- **Space キーではなく S キー**: `S` は当初「ソート」等の別ショートカット候補として一度予約されそうになった経緯があり ([[adr-0023-lightbox-random-shuffle]] の議論で言及)、汎用性の高い Space を選びました。Space は既存のライトボックスキーマップ（Esc / ←→ / F / R）と衝突しません。

## Status

承認済み

## Consequences

- **「拡大しながら削除候補をマーク → 閉じて一括削除」のフローが完結**しました。ユーザーは大量の画像を素早くスクリーニングでき、判断の質を落とさずに削除を進められます。特にバッチ生成 ([[adr-0002-batch-generation-sequential-loop]] / [[adr-0012-remove-batch-size-combination-cap]]) で一気に数十枚を作った後の選別作業で威力を発揮します。
- **選択 Set が単一情報源**として維持されるため、ライトボックスとギャラリータイルで同じアイテムを見たときに、選択状態が常に一致します。二重管理起因の不整合バグが構造的に発生しません。
- **`lightboxIndex === -1` のプレビューモードでは選択トグルが非表示**なので、ユーザーに「押しても意味がない」ボタンを見せずに済んでいます。Space キーも空振りしないため、誤動作の混乱もありません。
- **ライトボックス内直接削除を意図的に持たないポリシー**は、後の [[adr-0033-lightbox-eye-button-open-in-preview]] や [[adr-0034-gallery-d-key-delete-shortcut]] の設計判断にも継承されています。破壊的操作は必ず確認モーダル + 一覧見える状態で、というプロジェクト全体の原則になっています。
- **`resolveLightboxKey` のショートカット行が 1 個増えた**ため、[[adr-0013-lightbox-info-panel]] で述べた「既存のキーと衝突しない」チェックリストがまた 1 増えました（Esc / ←→ / F / R / Space の 5 個）。今後キーを追加する際は 5 個との衝突確認が必要です。
