# ADR 29: SD/SDXL アーキテクチャ差異は「トグルで単一情報源化 + LoRA はバッジのみ」で扱う

## Context

Stable Diffusion には少なくとも 2 系統のベースアーキテクチャ（SD1.5 系と SDXL 系）が並存し、それぞれで **推奨解像度が大きく異なります** (SD1.5 は 512² 中心、SDXL は 1024² 中心)。両者を混ぜて扱う UI は複数の落とし穴を抱えます。

- 解像度候補: SDXL 用の `1024×1024` を SD1.5 モデルで生成すると解像度過大で構図崩壊、逆に SD1.5 用の `512×512` を SDXL モデルで生成すると解像度過小でぼやける
- LoRA の互換性: SD1.5 向け LoRA は SDXL モデルには適合せず、逆も真
- バッチ生成 ([[adr-0002-batch-generation-sequential-loop]]) のモデル切替: SDXL のみでバッチ回したい / SD1.5 のみで回したい、というニーズが実運用で発生

[[adr-0003-sd-sdxl-model-detection-heuristic]] で「モデル名に `xl` が入っていれば SDXL」ヒューリスティックを、[[adr-0009-safetensors-header-sdxl-detection]] で `.safetensors` ヘッダー解析による正確な判定に置き換えてきました。しかし当初 ADR 9 は「SDXL モデルを UI 上から **除外** することでミスを予防」する設計でした。これは実運用で明らかに副作用が大きく、「SDXL モデルを持っているのに使えない」という状況を生んでいました。

Flux などの新種アーキテクチャも徐々に出現しており、`xl` ヘッダーヒューリスティックだけでは「不明 (unknown)」に分類されるモデルが混ざる状況になっていました。LoRA についても、SD の `/sdapi/v1/loras` レスポンスの `metadata['modelspec.architecture']` → `ss_base_model_version` の順にフォールバックしてアーキテクチャ判定を試みたところ、洋一郎さんの実データ 21 件中 8 件 (約 4 割) がメタデータ欠落で `unknown` になる現実がありました。

## Decision

**form の「SD / SDXL」セグメントトグルを単一情報源として導入し、SDXL モデルの UI からの除外を撤廃します。LoRA のアーキテクチャ互換性は「不一致が確実に分かる場合のみ ⚠ バッジ表示」に留め、選択肢からは除外しません。** 具体的な設計は次の通りです。

- **`modelTypeFilter: 'sd15' | 'sdxl'` トグル**: form 冒頭にセグメント UI を配置します。このトグルが次の 4 箇所を単一情報源として制御します。
  - モデルピッカー: 対応する type のモデルだけをドロップダウンに列挙
  - 解像度ピッカー: SDXL 用 ([[adr-0010-sdxl-ratio-orientation-size-preset]]) / SD1.5 用 ([[adr-0014-sd15-ratio-orientation-size-preset]]) を切り替え
  - バッチ「モデル切替」 ([[adr-0002-batch-generation-sequential-loop]]): 対応する type のモデルだけを巡回
  - バッチ「サイズの組合せ」: 対応する type のプリセット表からクロス積を生成
- **初期値の決定**: SD で今アクティブなチェックポイントの type から自動計算します。SDXL がロード中なら `'sdxl'`、そうでなければ `'sd15'`。ユーザーがトグルを操作するとその選択が優先されます。
- **不明モデル (Flux 等) は `'sd15'` 側に含める**: 3 値目 `'unknown'` は導入せず、`sd15` 側の「非 SDXL」バケットに寄せます。実運用で 3 値目を導入するほどの Flux モデル数がまだ多くなく、UI 複雑化のコストが上回るためです。
- **`GET /api/sd-loras` の拡張**: SD の loras レスポンスに `type: 'sd15'|'sdxl'|'unknown'` を付与して返します。`metadata['modelspec.architecture']` → `ss_base_model_version` の順にフォールバック判定します。チェックポイントと違い、ヘッダー再解析は不要で SD 側の API メタデータで足ります。
- **LoRA アーキテクチャ互換性は「バッジ表示のみ」**: form の LoRA セレクトで各項目の右に、`type` が `modelTypeFilter` と **確実に** 不一致な場合のみ `⚠(for SDXL)` / `⚠(for SD1.5)` バッジを添えます。`type: 'unknown'` は情報不足なので警告を出さず、選択肢からも除外しません。ユーザー自身の判断に委ねます。

代替案として次を比較検討し、いずれも却下しました。

- **SDXL モデルの UI 除外を継続 ([[adr-0009-safetensors-header-sdxl-detection]] の当初路線)**: 予防安全ですが、「SDXL を使いたい」という明確な意図を持ったユーザーの利用機会を奪います。実運用データが増えるにつれて、除外の副作用が予防の効果を上回りました。
- **モデルアーキテクチャに応じて form UI を丸ごと切り替える (トグルではなくラジオ)**: 実質同じですが、セグメントタブの方が場所を取らず、SDXL/SD1.5 切替が form の他 UI と統合されて見えるため、セグメントを選びました。
- **LoRA も互換性で除外**: SDXL モデルを選んだら SD1.5 用 LoRA を非表示にする案です。しかし LoRA 側の判定が 4 割の unknown 率という事実の前に、除外は誤検知で「使えるはずの LoRA が消えた」体験を生みます。バッジ警告に留めて **ユーザーの判断を尊重** する方針にしました。
- **`type: 'unknown'` を独立 3 値目としてトグルに追加**: Flux 対応が視野に入りますが、実データでは Flux モデル数が少なく、UI 複雑化のコストが利得を上回りました。将来の必要性に応じて再検討します。

## Status

承認済み

## Consequences

- **SDXL モデルが form から使えるように復活**しました。ユーザーが明示的にセグメントトグルで SDXL 側を選んで使う運用になり、アーキテクチャ混在起因の解像度不一致は「トグルを見れば分かる」構造的に予防されます。
- **単一情報源化**により、モデル / 解像度 / バッチ対象がすべて `modelTypeFilter` から derive されるため、片方だけ変えて片方が古い、という不整合が発生しません。トグルを切り替えると 4 箇所の候補が一斉に更新されます。
- **[[adr-0010-sdxl-ratio-orientation-size-preset]] と [[adr-0014-sd15-ratio-orientation-size-preset]]** の解像度ピッカーは、このトグルを既存前提として設計されています。両 ADR を後から読むと「なぜトグル前提で書かれているか」の背景がこの ADR にあります。
- **[[adr-0016-defer-sdxl-misclassification-fix]] のリモート SD 誤判定バグ**は本 ADR 決定によって回避策が明確になりました。ヒューリスティック誤判定が起きても、ユーザーがトグルを手動で正しい側に倒せば動くようになったためです。バグ自体の修正は依然として保留ですが、ユーザー側のリカバリー手段が確保された点で影響度は下がっています。
- **LoRA バッジ方針** により、⚠ が出ても「使うかどうか」の最終判断はユーザーに残ります。実運用では ⚠ が出た LoRA を意図的に使うケースは少ないですが、「Sumica が勝手に隠す」という主体性の奪取は起きません。unknown 率 4 割の現状に照らして、安全側志向 (「いきなり除外はちょっと心配」) を反映した設計です。
- **`GET /api/sd-loras` のレスポンス shape 変更**により、既存のクライアントは `l.type` を扱う実装に更新が必要でした。破壊的変更ですが、Sumica はサーバー・クライアント同一デプロイなので実質問題になっていません。
- **Flux 系新アーキテクチャの正式対応**は将来課題として残ります。SD3・Flux・Sana 等が広まると、2 値トグルでは足りなくなる可能性があります。3 値化のコスト vs 利得を再評価する時期がきたら、この ADR を supersede する新 ADR を書くことになります。→ [[adr-0042-flux-support-3way-architecture]] で 3-way 化を実装しました。本 ADR の「単一情報源」設計は N-way にそのまま拡張できる形で保たれています。
