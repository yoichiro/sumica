# ADR 31: API 接続設定はランタイム可変から `.env`-only へ縮退する

## Context

Sumica 初期には、ヘッダー右上に「⚙️ 設定」ボタンがあり、そこから開くダイアログで `LM_STUDIO_URL` / `STABLE_DIFFUSION_URL` / `LM_STUDIO_MODEL` の 3 つのランタイム変更が可能でした。実装としては、サーバー側に `let lmStudioUrl = ...` のように書き換え可能な module-level 変数を保持し、`POST /api/settings` エンドポイントで受け取った値でこれらを書き換える形でした。

この設計は「LM Studio や SD を別マシンで動かしていて、稼働中に接続先を切り替えたい」というユースケースを想定したものでしたが、実運用で以下の問題が明確になりました。

- **実際の利用パターン**: 洋一郎さんが Sumica を運用する中で、接続先は起動から終了までほぼ固定でした。設定ダイアログを開くのは初回セットアップ時と、ごく稀に SD をリスタートしたとき程度で、恒常的なランタイム変更のニーズは実質ありませんでした。
- **セキュリティ懸念**: `POST /api/settings` は認証なしで、CORS も緩いままでした。悪意ある Web ページから叩かれると接続先を任意 URL に切り替えられ、以降の LLM/SD 呼び出しがそちらに漏れる可能性がありました。CORS を [[adr-0001-client-side-firebase-persistence]] のポリシーでフロントエンド origin に限定はしているものの、そもそもこのエンドポイント自体を廃止する方が根本解決です。
- **状態管理の複雑化**: `let` の可変変数は、他の module-level `const` と混在すると読者にとって「これは const? let?」を毎回意識させます。設定変数だけ `let` になっている理由も docs に書かれておらず、認知負荷を生んでいました。
- **リーンさとの衝突**: Sumica の設計原則は「単一ローカルユーザー向けの薄いツール」で、複雑なランタイム設定管理は本質的にオーバーエンジニアリングでした。

方針転換として、選択肢は次の 2 つでした。

- **(A) クライアント UI のみ削除、サーバー側の可変変数と `POST /api/settings` は残す**: UI を隠すだけで、curl 等からは変更可能。セキュリティ懸念は残ります。
- **(B) 両方削除して `.env`-only にする**: ランタイム変更手段そのものを取り除き、設定変更したければ `.env` を書き換えてサーバー再起動する運用に統一。

## Decision

**(B) を採用します。クライアント UI・サーバーエンドポイント・ランタイム可変変数を **すべて削除** し、`LM_STUDIO_URL` / `STABLE_DIFFUSION_URL` / `LM_STUDIO_MODEL` は `server/.env` からブート時に `const` として一度読み込むだけの構成にします。** 具体的な設計は次の通りです。

- **クライアント側**: `AppHeader` の ⚙️ ボタンと、それが開いていた設定ダイアログコンポーネントを丸ごと削除します。
- **サーバー側**: `POST /api/settings` エンドポイントを削除します。`let lmStudioUrl = ...` などの module-level `let` を `const` に変更し、`.env` から `process.env.LM_STUDIO_URL` をブート時に一度読み込む形にします。
- **`GET /api/status`**: 現在の設定値を返す既存エンドポイントは残します。値は const になっただけで、読み取りは変わりません。UI 側でこの値を「現在の接続先」として表示する用途は継続します。
- **設定変更したい場合の運用**: `server/.env` を編集し、`npm run dev` を再起動します。CLAUDE.md にもこの手順を明示します。

代替案として次を比較検討し、却下しました。

- **(A) UI のみ削除、サーバーは残す**: 半端な選択で、セキュリティ懸念が解決しません。むしろ「UI がないのに API は生きている」という状態は、コードベースに詳しくない未来のコントリビュータを混乱させます。完全削除の方がクリーンです。
- **認証付き設定 API に強化**: 単一ローカルユーザー向けツールに認証を導入するのは重すぎます。認証の運用コスト (トークン管理、失効、パスワード忘れリカバリー) を負う価値がありません。
- **`.env` 読み取り + ランタイム変更両立**: 起動時は `.env`、ランタイム変更は API という two-source-of-truth 設計。混乱の温床です。一方に統一するべきです。

## Status

承認済み

## Consequences

- **セキュリティ攻撃面が減少**しました。設定変更を狙う攻撃ベクトルが存在しなくなり、CORS 制限の厳密性への依存も 1 段階緩みました ([[adr-0001-client-side-firebase-persistence]] の CORS ポリシー自体は継続)。
- **サーバーコードの module-level 変数がすべて `const`** になり、状態変化しないという保証が読者に伝わります。「これはランタイムで変わるのか？」という疑問がゼロになりました。
- **設定変更に再起動が必要になった**トレードオフがあります。ただし変更頻度が非常に低いため、実運用でこの再起動コストは無視できる範囲です。頻繁に接続先を切り替えたいユーザー（あまりいませんが）は、`.env` の値を編集して `npm run dev` を再起動するだけです。
- **CLAUDE.md の説明が簡潔化**しました。「`lmStudioUrl`, `stableDiffusionUrl`, `lmStudioModel` は `.env` からブート時に一度読み込む const。ランタイム変更エンドポイントはかつて存在したが削除された」という 1 文で完結します。歴史的経緯を残しつつも、現在の挙動は非常にシンプルに伝えられます。
- **将来の GUI 設定管理は復活しない前提**でこの ADR は書かれています。もし複数ユーザー対応や、SaaS 化などの方向に舵を切るときは、この ADR を supersede する新規 ADR を書くことになります。単一ローカルユーザー想定の Sumica 現状では、この方針が最もリーンで安全です。
- **削除された `POST /api/settings` の URL パス**が将来別用途で再利用される場合、この ADR を読んで「かつて何があったか」を確認できます。同名パスを別意味で再利用するのは避けたほうがよく、`POST /api/config` などの別名にするのが望ましいです。
