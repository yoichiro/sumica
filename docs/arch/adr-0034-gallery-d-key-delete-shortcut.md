# ADR 34: ギャラリー画面に D キーの削除ショートカットを追加し、テキスト入力フォーカスで無効化する

## Context

Sumica のキーボードショートカット群はこれまでライトボックス側に集中していました ([[adr-0013-lightbox-info-panel]] Esc、[[adr-0023-lightbox-random-shuffle]] R、[[adr-0027-lightbox-space-selection-toggle]] Space、[[adr-0033-lightbox-eye-button-open-in-preview]] は非設定、加えて既存の ←→ / F)。ライトボックスはモーダル的に画面全体を占有するため、テキスト入力欄が同時に存在しないという前提でキーバインドが素直に成立していました。

一方で **ギャラリー画面** はメインアプリのメイン画面で、`ControlPanel` の prompt textarea・数値入力欄・model セレクトなどのフォーム UI が **同一画面内に共存** しています。この画面でショートカットを設けるには、テキスト入力中にキーが誤動作しないための guard を注意深く設計する必要があります。

実運用のワークフローでは、[[adr-0027-lightbox-space-selection-toggle]] で拡大しながら削除候補にマークをつけた後、ライトボックスを閉じてギャラリー画面に戻り、ギャラリーツールバーの「削除」ボタンをマウスで押す、という操作が発生していました。マウスからキーボードに手を戻す一往復が発生します。しかも「マークして閉じてすぐ削除」の高頻度サイクルでは、この一往復が体感的に重くなります。

ショートカットキーの選択肢としては次があります。

- **Delete キー**: 直感的だが、macOS の Delete は Backspace 相当で、テキスト入力欄でも活発に使われます。誤動作の余地が大きい。
- **Backspace**: 同上の理由で除外。
- **D キー**: "Delete" の頭文字で mnemonic として自然。既存の R/F/Space/←→ とも衝突しない。
- **Cmd/Ctrl + D**: モディファイア付きは覚えるコストが高く、ブラウザのブックマーク追加と衝突。

D キーは自然な選択ですが、prompt textarea に「d」を入力するときに削除確認モーダルが誤起動する事故が明確にあり得ます。実際に洋一郎さんも初期実装のテストで textarea 入力中に D を押して「その日の全部が消えかけたら怖い」と発言していました。これを構造的に防ぐ guard が必須です。

同時に、削除確認モーダル自体は既存の `DeleteConfirmModal` が存在し、選択件数 (1 件以上) + `handleRequestDelete(Array.from(selectedIds))` の呼び出しで発火します。D キーはこの既存フローを呼び出すだけで、削除ロジック本体には触れません。

## Decision

**ギャラリータブが開いていて、選択が 1 件以上あり、ライトボックス・削除確認モーダルが開いておらず、テキスト入力欄にフォーカスが無いすべての条件を満たすとき、D/d キーで削除確認モーダルを開きます。** 具体的な設計は次の通りです。

- **キーバインド**: `window.addEventListener('keydown', ...)` の useEffect で `e.key === 'd' || 'D'` を判定します。`useEffect` の依存配列は `[rightTab, selectedIds, lightboxUrl, showDeleteConfirm]` にし、これらのいずれかが変わったらリスナーを張り直します。
- **5 段ガード**（この順で判定、いずれかで early return）:
  1. `rightTab === 'gallery'` でなければ何もしない（フォームタブ・プレビュータブでは効かせない）
  2. `selectedIds.size === 0` なら何もしない（選択ゼロなら削除対象がない）
  3. `lightboxUrl` があれば何もしない（ライトボックス独自の keymap 優先）
  4. `showDeleteConfirm` が true なら何もしない（モーダルが既に開いているときの二重発火防止）
  5. `e.target` が `INPUT` / `TEXTAREA` / `contentEditable` のいずれかなら何もしない（**これが本 ADR の中心的な設計判断**）
- **押下時の挙動**: `e.preventDefault()` してから `requestDelete(Array.from(selectedIds))` を呼びます。既存の削除ボタンクリックと **完全に同じ挙動** で、独自の削除フローは実装しません。
- **視覚ヒント**: ギャラリーツールバーの「削除」ボタンに `title={t.gallery.deleteHotkeyHint}` を追加し、ホバー時に「選択した画像を削除 (D)」/ "Delete selected images (D)" を表示します。

代替案として次を比較検討し、いずれも却下しました。

- **Delete / Backspace キー**: 前述の通り、テキスト入力欄での衝突が確実に発生します。D の方が安全です。
- **Cmd/Ctrl + D**: モディファイア付きは覚えるコストが高く、Chrome のブックマーク追加ショートカットとも衝突します。修飾なしの D で十分。
- **Delete ボタンにフォーカスがあるときだけ D を有効化**: ボタンにフォーカスを合わせる操作が発生するので、キーボード一直線のフローが崩れます。ガードは「テキスト入力欄にフォーカスがない」というプル型判定が最もリーン。
- **Delete モーダル内で D で確定**: 面白いが、モーダル内は既に Enter で確定できるので冗長です。
- **プロンプト textarea に IME 変換中の判定を追加**: `isComposing` チェックまでは加えていません。実運用では IME 変換中に「d」だけ確定して未確定入力にならないケースは稀で、Textarea フォーカス判定で実質十分でした。もし将来変換中誤起動の報告が出たら、`e.isComposing` チェックを追加する余地があります。

## Status

承認済み

## Consequences

- **ギャラリー整理フローがキーボード完結**しました。[[adr-0027-lightbox-space-selection-toggle]] の Space で選択マーク → 閉じる → D で削除確認 → Enter で確定、というフローが完全にキーボードで済み、マウスに手を戻す必要が消えました。
- **テキスト入力欄フォーカス時の誤動作が構造的に防がれています**。特にプロンプト textarea に「d」（"draw" とか "digital art" とか）を入力するときに削除が起動しないため、洋一郎さんが懸念していた「その日の全部が消える事故」は発生しません。
- **既存の削除フロー (`requestDelete`) を再利用**しているため、D キー経由と「削除」ボタン経由の挙動が **完全に一致** します。将来 `requestDelete` の内部実装を変更しても、両経路が同時に反映されます。バグの発生源が二重にならない設計です。
- **キーボードショートカットが Sumica 全体で 6 個目** (Esc / ←→ / F / R / Space / D) になりました。ライトボックス側 (Esc / ←→ / F / R / Space) 5 個 + ギャラリー側 (D) 1 個で、混同しない前提で追加できました。今後キーを追加する場合、6 個との衝突確認が必要です。
- **[[adr-0027-lightbox-space-selection-toggle]] と組み合わせた「拡大しながら削除候補マーク → 閉じて D で削除」ワークフロー**は、実運用でパワーユーザー向けの最速削除フローとして機能しています。特に SD の高速生成環境で数十〜100 枚単位の一括生成をする場合、この整理フローの速さが体感の違いに直結します。
- **将来テキスト入力欄が増えても、5 段ガードのうち INPUT/TEXTAREA/contentEditable 判定が自動的にカバー**します。新しい入力欄を追加する際に、この ADR を思い出して個別の guard を仕込む必要はありません（フォーカスした要素の tagName で判定しているため）。
- **IME 変換中の判定は現状追加していない**ため、日本語入力中に「d」だけ確定してすぐ Enter を押す、といった特異な操作をされた場合、誤動作の余地は理論上ゼロではありません。実運用で報告が出たら `e.isComposing` チェックを追加する運用です。
