# ADR 35: ランキングの並び順を Wilson 下限から「お気に入り絶対数」に変更する

## Context

[[adr-0021-favorite-recipe-rollup-ranking]] で導入したお気に入りレシピランキングは、当初 Wilson 95% 信頼区間下限 (Wilson lower bound) の降順でレシピを並べていました。統計的には「試行回数を考慮しつつ fav 率の下限を推定する」という妥当な選択で、1/1 のまぐれ当たりが 20/25 の実績あるレシピを上回らないという性質を持ちます。

しかし [[adr-0024-ranking-recipe-full-form-restore]] で recipe 粒度が 8 次元から 15 次元 + LoRA weight に拡張された後、洋一郎さんの実運用データを見返すと次のことが明らかになりました。

- Sumica のワークフローは「どんな画像が出るか分からないから **あてずっぽうで多量に生成** し、その中から気に入ったものだけを ⭐ でマークする」という粘り強い試行 → 選別型
- この運用では `total` (試行回数) は「そのレシピをどれだけ試したか」= **ユーザーの粘り強さの指標** であって、レシピ本来の魅力を測るものではない
- 逆に `favs` (お気に入り数) は「そのレシピで何枚のキーパーが得られたか」を直接示す指標になる

洋一郎さん自身の言葉として:

> 生成数は「どんなものが生成されるかわからない」から「あてずっぽうで多量に生成する」ことが多いために「生成数はランキングにおいて何の指標にもならない」と思っています。単純に「お気に入りになった画像の本数」がレシピの良さの序列なんじゃないかと思っています。

Wilson 下限は「試行回数の少ないレシピの信頼性を割り引く」機能を持ちますが、Sumica ではその「試行回数の多寡」がユーザーの意思ではなく **たまたま何回生成したか** で決まっています。統計的には正しくても、Sumica のユースケースでは意味を失っていました。

加えて、[[adr-0024-ranking-recipe-full-form-restore]] の副作用として recipe 数が 456 → 1079 (2.4 倍) に増え、多くのレシピが試行回数 1〜2 に留まる状況になりました。これは Wilson 下限が正しく機能するには試行数が少なすぎるレンジで、Wilson の統計的なメリットが実質的に得られなくなっていたことも示唆します。

## Decision

**ランキングの並び順を Wilson 下限降順から「お気に入り絶対数 (`favs`) の降順」に変更します。tie-break は `updatedAt` 降順で、最近 fav が追加されたレシピを上位に表示します。フィルタは `total >= minSample` から `favs >= 1` に変更し、`minSample` パラメータは廃止します。** 具体的な設計は次の通りです。

- **`rankRecipes()` の書き換え**: `client/src/utils/rankingAnalysis.ts` を次のように変更:
  ```ts
  export function rankRecipes(rollups: RankingRollup[], topN = 10): RankedRecipe[] {
    return rollups
      .filter((r) => r.favs >= 1)
      .slice()
      .sort((a, b) => b.favs - a.favs || b.updatedAt - a.updatedAt)
      .slice(0, topN);
  }
  ```
- **`wilsonLower()` 関数の完全削除**: 使わなくなるため、関数ごと `rankingAnalysis.ts` から取り除きます。`RankedRecipe` 型からも `wilson` / `rate` フィールドを削除し、`RankingRollup` のシェイプそのままを alias します。
- **`RankingPanel.tsx` の `minSample` prop 削除**: props インターフェースから外し、既定値の再現も不要にします。
- **i18n key の削除**: `t.ranking.headerWilson` / `headerRate` / `favsShort` を両言語から取り除きます。UI からも Wilson% / Rate% の表示は完全に消えます。
- **tie-break が `updatedAt` 降順である理由**: fav 数が同じレシピが並んだとき、**最近 fav が追加されたレシピを上に持ってくる** ことで、「今の気分」に沿ったレシピが優先されます。過去の一時的なブームで fav が付いた古いレシピが延々と 1 位のままになるのを防ぎます。

代替案として次を比較検討し、いずれも却下しました。

- **Wilson 下限を維持したまま表示ラベルだけを変える**: ユーザー視点で「fav 数が表示されているのに並び順は違う」というギャップが混乱を生みます。表示と並び順が完全に一致するのが直感的です。
- **fav 数 desc + tie-break を hash 辞書順**: 決定的な順序を得られますが、`updatedAt` desc の方が「最近のアクティビティ」を反映して情報量が多い選択です。実質ランダムに近い hash 順よりユーザー価値が高い。
- **fav 数 desc + tie-break を fav 率 (favs/total) desc**: fav 数と fav 率の両方を見る合成指標ですが、そもそも total を評価軸に入れないという本 ADR の趣旨と衝突します。Sumica の実運用で total を評価軸にしない、という明確な意思決定を維持します。
- **fav >= 1 ではなく fav >= 2 のフィルタ**: 「1 発当たり」を除外する予防ですが、洋一郎さんの実データでは fav 1 のレシピが多数存在し、そこには意図的にお気に入りに登録されたキーパーも含まれます。恣意的な閾値でユーザーの意思を切り捨てるより、fav = 0 のレシピだけを除外するミニマル方針が妥当です。
- **`wilsonLower()` を将来オプションとして残す**: A/B テストや代替ソートオプションを提供する余地はありますが、コードベースにデッドコードを残す判断ではありません。必要になれば git 履歴 (この ADR 実装のコミット直前) から復元できます。

## Status

承認済み

## Consequences

- **表示と並び順が一致**するようになり、ユーザーが「fav 数が多いレシピが上」という直感的なランキングを得られます。「なぜこの順?」という疑問がゼロになります。
- **fav = 0 のレシピが除外**されたことで、ランキング表示が「実際に価値を提供したレシピ」だけに絞られます。洋一郎さんの実データでは fav = 0 の rollup がほぼ全数を占めていたため、可視化される件数が劇的に減少しますが、これは「意味のある候補だけを見せる」という UX 改善です。
- **`wilsonLower()` / `RankedRecipe.wilson` / `RankedRecipe.rate` / `headerWilson` / `headerRate` / `favsShort` が完全消滅**しました。統計的な複雑さが取り除かれ、`rankingAnalysis.ts` は 30 行余りの純粋な sort ロジックだけになりました。可読性・保守性が向上しています。
- **[[adr-0021-favorite-recipe-rollup-ranking]] の当初根拠 (Wilson で試行数の少ない当たりを割り引く)** は、Sumica のワークフローの実態を経て意味を失いました。ADR 21 の Status セクションに本 ADR による Amended を追記します。
- **`minSample` の閾値がなくなった副作用として、favs=1 のレシピも上位に表示され得ます**。まぐれ当たりが上位を占めるリスクは理論上残りますが、洋一郎さんの主張は「たった 1 回でも ⭐ したなら、それはユーザーの明確な意思」であり、恣意的な閾値でその意思を切り捨てないという原則を採択しています。
- **tie-break を `updatedAt` desc にしたため、同じ fav 数のレシピは「直近 fav が追加された順」で並びます**。ユーザーが「最近気に入ったスタイル」を目にする機会が増え、埋もれたレシピが再発見されにくくなる副作用もありますが、実運用では発見系のフローとして [[adr-0023-lightbox-random-shuffle]] のランダムシャッフルが別途あるため、双方で補完できています。
- **[[adr-0024-ranking-recipe-full-form-restore]] で recipe 数が 2.4 倍に増えた副作用** はランキング表示側でも顕在化していましたが、本 ADR の `favs >= 1` フィルタで自然と絞られる形になっており、粒度細分化のトレードオフが緩和されています。
