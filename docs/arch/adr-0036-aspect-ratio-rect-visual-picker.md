# ADR 36: AspectRatioRect による視覚的アスペクト比プレビュー

## Context

Sumica のフォームでは、[[adr-0010-sdxl-ratio-orientation-size-preset]] と [[adr-0014-sd15-ratio-orientation-size-preset]] で採用したプリセットピッカー方式により、アスペクト比を `1:1` / `4:3` / `9:7` / `3:2` / `16:9` / `21:9` / `3:1`（SDXL）や `1:1` / `5:4` / `4:3` / `3:2` / `5:3` / `16:9` / `2:1`（SD1.5）といったラベルつきのチップボタンとして表示していました。ラベルの文字列は正確ですが、`21:9` と `16:9` が並んで見えたときに実際にどのくらい横長かの差が瞬時に読み取れず、洋一郎さんは「`3:2` と `16:9` の見分けは頭の中でしかできない」と感じていました。同様に [[adr-0021-favorite-recipe-rollup-ranking]] で導入したランキング行にも `1024×768 (4:3)` のようなテキスト表記しかなく、複数レシピの縦横形状を並べて比較する場面では視認性が乏しい状態でした。

一方、ランキング行と 2 箇所のアスペクト比ピッカー（SDXL 用 / SD1.5 用）は、同じ「W×H から比を割り出して視覚化する」ロジックを必要としており、それぞれで別実装を持たせるとスタイルの一貫性が崩れやすいという懸念もありました。orientation（縦・横）の切り替えは既に別 UI（`selectedOrientation` / `selectedSd15Orientation`）で管理していますが、アスペクト比ラベル `4:3` は常に landscape 前提の表記のため、portrait を選んだ瞬間に「見た目上の縦横」がラベルと逆転する認知の齟齬もありました。

## Decision

**アスペクト比を反映した矩形を描画する共通コンポーネント `client/src/components/AspectRatioRect.tsx` を新設し、フォームの SDXL / SD1.5 アスペクト比ピッカーとランキング行の両方で再利用します。** 描画は SVG ではなく単一の `<div>` を CSS で W×H スケールする実装とし、`maxEdge` / `borderColor` / `background` / `borderWidth` を props で受けます。長辺が常に `maxEdge` に収まるようにもう一方を比例縮小することで、極端に横長・縦長な比でも枠内に収まります。

矩形の縦横は、フォームピッカーで active になっている orientation に追従します。`selectedOrientation === 'portrait'` の場合は W と H を swap して描画し、landscape の場合はプリセットの生値（landscape 前提の W×H）をそのまま使います。`1:1` は swap しても形状が変わらないため、orientation UI 自体が非表示のまま整合します。ランキング行では recipe 側に既に確定した `width × height` が入っているため、生値をそのまま渡すだけで自然に正しい向きになります。

代替案として次を比較検討し、いずれも却下しました。

- **矩形を SVG 要素として描画する**: 標準的な選択肢ですが、Sumica では既存 UI がすべて `<div>` + CSS で組まれており、SVG を持ち込むと ID 衝突回避や viewBox の管理などのボイラープレートが増えます。矩形一つに `width` / `height` / `border` を指定するだけなら `<div>` で十分で、実装量が最小になります。
- **アスペクト比ごとに専用のアイコン（PNG / SVG）を用意する**: デザイン的にはリッチにできますが、7 種類 × orientation 2 種類 = 14 種類のアセットを維持する必要があり、比を追加するたびにデザイン作業が発生します。動的に矩形を計算する方式なら、`SDXL_PRESETS` / `SD15_PRESETS` に新しい比を足すだけで自動追従します。
- **共通コンポーネントにせず、フォーム側とランキング側で別実装を持たせる**: 初期は簡単ですが、ボーダー色・背景色・maxEdge の一貫性を保つのに人手が必要になります。共通化して props で外観差分を吸収する方が、将来の一貫性維持コストが下がります。
- **フォーム側だけ視覚化してランキング側はテキスト維持**: 洋一郎さんは「複数レシピの縦横形状を並べて比較したい」というユースケースをランキング側でも持っていたため、両方に入れて情報密度を揃える方が有意義です。

## Status

承認済み

## Consequences

- **視覚的比較が速くなりました**。SDXL 側の `21:9` / `16:9` / `3:2` の 3 チップを並べたときに、実際の横長度合いが 3 段階でひと目でわかるようになりました。ランキング行では、上位レシピの縦横形状を上下に並べたときに「縦長系ばかり気に入ってる」「横長系はほとんどない」といったパターンが視認しやすくなっています。
- **フォームピッカーとランキング行のスタイル差分は props だけで吸収**できるようになりました。フォームでは active ボタン内で白ボーダー + 半透明白背景（`borderColor="#fff"`, `background="rgba(255,255,255,0.22)"`）、非 active では青ボーダー + 淡青背景を使い分けています。ランキング行はデフォルト値のまま使えており、共通化の恩恵を受けています。
- **orientation 連動により、`portrait` に切り替えた瞬間に矩形もその場で縦向きに回転**します。これは前述の「ラベル `4:3` は landscape 表記なのに実効は縦」という認知齟齬を、ラベル側ではなく矩形側の追従で解消しています。ラベル自体は landscape 表記のまま（変更しない）としており、比の正規名（4:3 が常にそう呼ばれる）を尊重した判断です。
- **`AspectRatioRect` は現在 `ControlPanel` の SDXL/SD1.5 ピッカーと `RankingPanel` の 2 箇所で利用**されています。今後 [[adr-0030-favorite-images-foundation]] 系の別ビュー（例: ライトボックスの info パネル）にも比のプレビューを足したくなった際、同じコンポーネントを import するだけで済みます。
- **軽微な副作用として、フォームの比チップに矩形が追加された分だけ横幅を消費する**ようになりました。既存の狭幅ビュー向けレイアウトは `flex-wrap: wrap` で自動的に折り返されるため実質影響はありませんが、将来モバイル向けに矩形サイズをさらに詰めたくなる可能性はあります。その場合は `maxEdge` を小さくして呼び出せば済みます。
- **`gcd` + `parseSize` によるランキング行側のサイズ解析**は `RankingPanel.tsx` にローカル関数として維持しています。ランキング固有の「WxH 文字列 → 数値対」変換であり `AspectRatioRect` の責務ではないため、共通コンポーネントには含めていません。将来別コンポーネントでも WxH 文字列パースが必要になったら、別ユーティリティに切り出す判断を再検討します。
