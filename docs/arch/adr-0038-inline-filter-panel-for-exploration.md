# ADR 38: 探索的フィルタリング用のインライン展開パネル

## Context

[[adr-0037-history-gallery-filter-surface]] で追加した多軸フィルタは、当初「フィルターボタン → ポップオーバー展開 → 条件を決めて閉じる」というモーダル的な UI で実装しました。BatchGenerationModal と同じ View Transition パターン（`view-transition-name: gallery-filter-morph` を button と popover で共有）でモーフィングも入れ、見た目としてはリッチな仕上がりでした。

しかし実運用で試してみたところ、フィルタ機能の使い方は「一度条件を決めて閉じる」よりも **「条件を軽く変えながらグリッド結果を見比べる」試行錯誤のワークフロー** であることが判明しました。ラジオを切り替えるたびにグリッドが動的に絞り込まれ、ユーザーはその変化を眺めながら次の絞り込みを考えるという使い方です。

このモードでポップオーバー UI は複数の摩擦を生みます：

- **結果が隠れる**：ポップオーバーがグリッドの一部を覆うため、フィルタ変更中の結果を確認しにくい。
- **開閉サイクルの摩擦**：「開く → 変更 → 閉じる → 見る → また変えたい → 開く」という繰り返しがリズムを壊す。
- **モーダル思考の押し付け**：ポップオーバーの UI 語彙は「決めて確定する」ものなので、「軽く試して戻す」感覚と衝突する。

洋一郎さんから「フィルターのための UI は、フィルター条件を変化させてどのような画像が列挙されるかを試行錯誤することに特化したものである必要がありそう」という洞察が挙がり、UI 語彙そのものを見直す判断に至りました。

## Decision

**ポップオーバーを廃止し、ツールバー行の直下に常時展開可能なインラインパネルとしてフィルタ UI を配置します。** トグルボタン自体は残しますが、押下時にパネルが上からスライドダウンで inline 展開し、グリッドを覆わずに押し下げる形にします。

具体的な設計は次のとおりです。

- **コンポーネント分割**：`GalleryFiltersPopover.tsx` を 2 export に分割します（`GalleryFilterToggleButton` と `GalleryFilterPanel`）。トグルはツールバー行の中に、パネルは行の下（外側）にレンダリングします。open state は HistoryGallery に lift し、両コンポーネントに props で渡します。
- **View Transition の継続**：ポップオーバー時代から使っていた `view-transition-name: gallery-filter-morph` の button ⇔ panel morph はそのまま継続します。button 側は「open のとき undefined、closed のとき `gallery-filter-morph`」で name を drop、パネル側は常時 `gallery-filter-morph` を保持することで、ブラウザに 1 スナップショットあたり 1 オーナーを保証します。`flushSync` で React の state コミットを同期化し、close 時にもアニメーションが飛ばずに実行されます（React 18 default batching だけでは new snapshot が old と同一に見えて 40ms 程度でアニメが打ち切られる問題を回避）。
- **Border-only VT target**：ボタン全体をトランジションの対象にすると、ラベル文字までパネル矩形へ引き伸ばされて視覚的に破綻します。ボタンの中に `<span aria-hidden position:absolute inset:0>` を敷き、これだけに `view-transition-name` を持たせることで、**枠だけがモーフィング**し、ラベルテキストは静止する挙動にしています。
- **常時 sticky**：ツールバー行とフィルタパネルをまとめて sticky ラッパーに入れ、グリッドをスクロールしてもフィルタ状態が常に見える位置に留まります。試行錯誤中の視認性を優先しました。
- **レイアウト**：パネル内は `flex + gap + flex-wrap` の 1 行横並びを基本とし、モデル種別（radio）・アスペクト比（select）・向き（radio）・モデル（select）・サンプラー（select）・クリアボタンの順で配置します。ラジオ（3 択以下）と select（可変数）を混在させることで、選択肢数に応じた最適な UI 語彙を軸ごとに選んでいます。
- **選択操作の統一**：閉じるボタンはパネル内から削除し、代わりに Escape キーとトグルボタン再クリックで閉じられるようにします。パネルが inline なので「外側クリックで閉じる」は逆に鬱陶しい（グリッドタイルをクリックしたら閉じてしまう）ため、削除しました。
- **フィルタボタンの位置**：トグルボタンは日付フィルタの **左** に置きます。ボタン → パネル展開のときにパネルが右方向に伸びるため、button が右寄りだとパネルがビューポート外にはみ出して横スクロールバーが出るからです。お気に入りボタンは変わらず右端に置いて親指到達しやすさをキープ。

代替案として次を比較検討し、いずれも却下しました。

- **常時展開のパネル**（トグルボタンなし）：探索モードには最適ですが、フィルタ不使用時（単に日付を眺めるだけの時）にも縦スペースを常に占有するのが冗長。「使うときだけ展開」のトグルを残す方が良い妥協点と判断しました。
- **左パネルに「🔍 フィルター」タブを追加**（フォーム / ランキングと並ぶ 3 番目のタブ）：ギャラリーの全高を使えて広いですが、左タブ切替の手間が発生し、右のグリッドを見ながらフィルタを操作しにくくなります。inline パネルの方が「グリッドとの近さ」で優位です。
- **ポップオーバーのまま位置調整だけ**：フィルタボタンを左端に置けばポップオーバーはビューポート内に収まりますが、「グリッドを覆う」問題は解決しません。UI 語彙自体を変える必要がありました。

## Status

承認済み

## Consequences

- **試行錯誤のリズムが生まれた**：ラジオを 1 つクリック → グリッドが変化 → 次の絞り込みを検討、というテンポで作業できるようになりました。開閉ボタンを押す動作がなくなり、フィルタ操作を「動作」ではなく「状態への微調整」として体験できます。
- **View Transition のパターン再利用**：`gallery-filter-morph` の name 共有パターンは、[[adr-0025-lightbox-view-transitions]] 系の view-transition-name 共有パターン (batch modal も同手法) と同じ手法をほぼそのまま流用できました。React 18 での `flushSync` 必須という細かい落とし穴もこの過程で明らかになり、ドキュメント化されました。
- **Border-only VT の副次的知見**：View Transitions が element ごとに snapshot を撮る仕様を利用し、「見た目上は 1 個の button に見えるが、内部的には枠だけがトランジション対象」という構造を作れることが分かりました。ラベル文字を静止させたまま枠だけを morph させることで、「文字がぐにゃっと伸びる」ような破綻を回避できます。今後別の UI でも同パターンが応用できます。
- **画面上の縦スペース消費**：パネルが inline 展開する分、グリッドが下に押し下がります。トグルボタンで畳めるためユーザー制御可能ですが、ノートパソコンなど縦解像度が狭い環境では 1 行分のグリッドが見切れる可能性があります。今後モバイル対応が本格化した際には、パネルの高さをさらに詰めるか、モバイル時はドロワー式にするかを再検討する余地があります。
- **`閉じる` ボタン削除の是非**：明示的な「閉じる」ボタンがなくなり、Escape キーとトグル再クリックの 2 経路になりました。キーボードユーザーには問題ありませんが、初見のマウスユーザーが「どうやって閉じるか」を戸惑う可能性は残ります。トグルボタンが常に見えていれば発見的に閉じられる想定ですが、ユーザビリティテストで問題が出れば "×" アイコンをパネル右上に追加する判断はあり得ます。
- **フィルタ選択の永続性はセッション内のみ**：[[adr-0037-history-gallery-filter-surface]] で決めた通り、localStorage には保存しません。試行錯誤の道具として、「白紙から始める」方が「前回の状態から始まる」よりも探索的意図に合致します。
- **filter button の位置移動の副次的効果**：フィルタボタンを日付の左に移動させたことで、popover 時代にあった「右にはみ出して横スクロールバーが出る」問題は根本的に消えました。inline パネル化した今となっては position の制約は関係ありませんが、button の並び「フィルター → 日付 → お気に入り → 件数」という視覚的グルーピングは、UI の意味論として「アクション → 条件表示」の順に並んだ結果として自然になっています。
