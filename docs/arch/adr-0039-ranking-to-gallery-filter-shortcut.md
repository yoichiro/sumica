# ADR 39: ランキングレシピからギャラリーフィルタへのショートカット

## Context

[[adr-0021-favorite-recipe-rollup-ranking]] で追加したランキングタブは、⭐ 数の多いレシピを Top-N で提示します。[[adr-0024-ranking-recipe-full-form-restore]] の「フォームに適用」ボタンで、そのレシピを生成フォームに書き戻して「もう一度似た画像を生成する」動線は作られていました。

しかし [[adr-0037-history-gallery-filter-surface]] で多軸フィルタを導入したことで、新しい動線が意味を持つようになりました。「このレシピで **過去に生成した** 画像を全部俯瞰したい」というユースケースです。ランキング行を見て「6件のお気に入り」と書いてあっても、実際にその 6 枚を並べて見比べる導線がありませんでした。

さらに、[[adr-0037-history-gallery-filter-surface]] のフィルタは日付＋お気に入り base scope の後段に位置するため、「全期間から特定条件で探す」を実現するには favoritesOnly を ON にする必要があります（date filter をバイパスする唯一の手段）。レシピは元々 favorites から抽出されているため、この scope は本質的に整合します。

## Decision

**ランキング各行に虫眼鏡ではなく `Filter` アイコン（漏斗）のセカンダリボタンを追加し、押下時にそのレシピの各パラメータをギャラリーフィルタに反映すると同時に、右ペインをギャラリータブに自動切替、さらに `favoritesOnly=true` を強制的に ON にする挙動を実装します。**

具体的な設計は次のとおりです。

- **ボタン配置とアイコン**：既存の「フォームにロード」ボタンをアイコンのみ（`RotateCcw`）に縮小し、その右隣に新しい「ギャラリーで探す」ボタン（`Filter` アイコン）を並置します。青塗り／青枠で視覚的に区別し、tooltip と aria-label で意味を提供。両者は 28×28px の正方形で並びます。`Filter` アイコンはギャラリー側のフィルタトグルボタンと同じアイコンで、「フィルタ関連の操作」であることを視覚的に伝えます。
- **パラメータ→フィルタ軸のマッピング**：`applyRecipeToGalleryFilter(recipe)` in App.tsx が recipe.params の各値を galleryFilters の 5 軸に写します。`arch` は `inferSdArchitectureFromTitle(rp.model, sdModels)`、`model` は stripped title、`sampler` はそのまま、`aspectRatio` は `computeAspectRatio(w, h)`、`orientation` は W×H 比較（正方形は null）。
- **favoritesOnly=true の強制**：`setFavoritesOnly(true)` を同一バッチで呼びます。これによりランキングの元データである全期間 favorites が base scope になり、フィルタが実データとマッチする状態が保証されます。
- **右ペインのタブ自動切替**：`setRightTab('gallery')` を呼び、ランキングを見ていたユーザーがそのままギャラリーの結果を目にできるようにします。左ペインのランキングタブは触りません（並べて見比べられる状態を維持）。
- **トースト通知**：「レシピをギャラリーフィルターに反映しました 🔍」を表示。フォーム側の「レシピをフォームに読み込みました 🎨」と対になっています。

代替案として次を比較検討し、いずれも却下しました。

- **favoritesOnly を触らない**：現在の date filter のまま base scope を維持する方針。しかしランキングは長期間にわたるお気に入り統計から出るため、レシピが 6 favs 持っていても「今日の履歴」に該当画像がない場合がほとんど。date filter のまま filter を適用すると、`filterOptions` が空になり [[adr-0037-history-gallery-filter-surface]] の stale-clear effects が全ての filter 値をリセットしてしまい、実質的に「モデル種別だけ有効・0件」になります。これは強制 favoritesOnly の必要性を裏付ける根拠でもあります。
- **date filter を「なし」にする独立モード**：ギャラリー全体を全期間ビューにするモードを追加する案。しかし現在の設計では `filterDate` は常に特定日に固定され、favoritesOnly が唯一の「日付をバイパス」経路です。「なし」を導入すると date filter の意味論が二段構造になり複雑化します。既存の favoritesOnly バイパスを活用する方が既存パターンとの整合性が高い。
- **アイコンとして虫眼鏡 (`Search`)**：初期実装では虫眼鏡アイコンを使いましたが、洋一郎さんから「フィルター条件に適用するので、フィルターのアイコンの方が良いのでは？」との指摘を受けて `Filter` に変更しました。ギャラリー側のフィルタトグルと同じアイコンを共有することで、「フィルタ関連」の視覚的関連性が強調されます。
- **1 ボタンで両モード対応**（例：click は form、Alt+click は gallery filter）：モードキーの発見性が低く、ランキング画面での主要な操作 2 つが同格の重要さを持つため、2 つの独立したボタンとして並列に見せる方が明快です。

## Status

承認済み

## Consequences

- **ランキングの「見て確認する」動線が成立**：ランキング行を見て「気になる」と思ったレシピについて、1 クリックで「実際にこれで生成された過去のお気に入り画像群」を並べて見比べられるようになりました。ランキング統計と実画像を結ぶループが閉じ、レシピの「良さ」を統計だけでなく実物で確認できます。
- **強制 favoritesOnly の副作用**：click 時にユーザーが元々 favoritesOnly=false で使っていた場合でも自動的に ON になります。ユーザーが「フィルタを解除しようとしたら favorites-only もまだ ON だった」で戸惑う可能性はあります。ただし ⭐ ボタンで即座に解除可能で、Toast にも「ギャラリーフィルターに反映」と明示されているため、混乱は最小化されると判断しました。
- **stale-clear の empty-guard との相互作用**：[[adr-0037-history-gallery-filter-surface]] で導入した `length > 0` ガードがない場合、この機能はワンクリックで「モデル種別だけ有効・他は空」の壊れた状態になってしまい、二度目のクリックで初めて全フィルタが適用されるという再現性のある不具合を持っていました。この不具合の解決は empty-guard の実装が必要条件で、両 ADR は互いに依存しています。
- **アイコン統一の副次的効果**：ランキング側の「ギャラリーで探す」ボタンとギャラリー側の「フィルタトグル」ボタンが同じ `Filter` アイコンを使うことで、「フィルタ関連の操作は共通のアイコン」という語彙が確立されました。将来別の場所で「フィルタに関わる操作」を足す場合、この語彙を継続することで発見性が上がります。
- **左右ペインの独立性**：左ペインの「ランキング / フォーム」タブは触らないため、ユーザーはランキングを見ながら右ペインでフィルタ結果を確認できる並列レイアウトになります。これは [[adr-0021-favorite-recipe-rollup-ranking]] のタブ設計（左＝入力パネル、右＝出力パネル）と整合します。
- **ユーザーが recipe と favorites の関係性を暗黙に学ぶ**：この機能を使うと「ランキングに載っているレシピの favorites は、当該レシピで生成した過去画像から来ている」という因果を体験的に理解できます。単なる統計表として見えていたランキングが、「⭐ を積み上げたら勝手にランキングされる仕組み」であることが体感で伝わります。
- **将来のフィルタ軸追加時の連鎖影響**：もし [[adr-0037-history-gallery-filter-surface]] に新しい軸（Scheduler / Hires / Refiner など）を追加する場合、`applyRecipeToGalleryFilter` にも同軸のマッピングを追加する必要があります。忘れると「レシピは Scheduler が Karras なのにフィルタ側で反映されない」という抜けが発生します。両所を同時に更新するチェックリストが今後必要。
