# ADR 41: スライドショー間隔のユーザー選択とカウントダウンリング表示

## Context

[[adr-0040-lightbox-slideshow-and-random-mode-toggle]] でライトボックスにスライドショー機能を導入した際、間隔は 5 秒固定として実装しました。ブレスト時点で「一般的な間隔・実装 UI ともにシンプル・将来変えたくなったらコードの定数を 1 行変えるだけ」という判断で、UI で選べる選択肢は明示的に out-of-scope に置いた経緯があります。

しかし実際に機能を触ってみた結果、洋一郎さんから「スライドショーの画像切り替えの間隔をやはり選択できるようにしたい」というフォローアップ要望が数時間後に上がりました。ユースケースとしては、5 秒だと「じっくり見る」用途には短すぎ、逆に長時間動画的な使い方をしたい場合に固定 5 秒では合わない、という体感差でした。

間隔選択の UI 設計には複数の選択肢があります。

- **常時見える離散プリセットのチップ列**: ライトボックス上部のツールバーに `5s / 15s / 30s / 60s` を常に並べる。発見性は高いが、ツールバーの水平方向を圧迫します。[[adr-0040-lightbox-slideshow-and-random-mode-toggle]] で既に Eye ボタンを退避させて狭くなっており、これ以上のボタン追加は避けたい状況でした。
- **ドロップダウン展開型**: 現在値のバッジをクリックしたらプリセットが縦に開く。UI 面積は節約できるが、開閉状態を管理する state と閉じる条件（outside click / Esc）が増えます。
- **連続値スライダー**: 1〜30 秒の範囲でスライダー。細かな調整は可能ですが、離散プリセット（5 段階前後）でも実用上十分で、スライダーの UI 面積はより大きくなります。
- **右クリックで循環**: Slideshow ボタンを右クリックするたびに次のプリセットに循環し、現在値はボタン上に小さくバッジ表示。UI の増加がゼロで、ミニマリスト方針と整合しますが、右クリックという操作は発見性が低いという弱点があります。

洋一郎さんは 4 番目の「右クリックで循環」を明確に選ばれました。UI をこれ以上増やしたくない、というライトボックスの絶対配置ツールバーの窮屈さへの配慮が背景にあります。発見性の弱さは、ボタンの tooltip 文字列に「右クリックで間隔を切替」というヒントを常時添えることで補います。

プリセット値の選択肢についても議論があり、`5s / 15s / 30s / 60s` が採用されました。当初私が提案した `2s / 3s / 5s / 10s / 15s` は「短めで細かく」寄りで、洋一郎さんが選んだ `5s / 15s / 30s / 60s` は「じっくり見る」寄りです。プリセット段階数は 4 個で、右クリック循環の 1 周が短く、覚えやすい水準になっています。

永続化については、選んだ間隔がリロード後にリセットされると毎回選び直しになるため、体感が悪化します。ブラウザ側で永続化する手段は 2 つあり、`localStorage`（サーバーレス、単一ブラウザ内でのみ有効）と Firestore（サインインしていれば端末横断で残る）です。この機能は「ライトボックスの再生ペースというごく個人的な好み」で、Firestore の書き込みコストや依存を持ち込むほどの重みはありません。`localStorage` で十分と判断しました。

さらに追加要望として、洋一郎さんから「切り替えまでの残り時間をスライドショーボタンの円周のボーダーで表現できないか」というリクエストが上がりました。動作中のインジケーターが欲しい、というのは触ってみて明らかに欲しくなる情報で、間隔可変化と同じセッション内で実装することにしました。ボタン自体は円形（`borderRadius: 50%`）なので、円周上にストロークを描画する SVG オーバーレイを重ねるのが自然です。CSS の `conic-gradient` で回転させる案もありましたが、SVG `<circle>` の `stroke-dasharray` / `stroke-dashoffset` の方が制御が細かく、方向・色・線幅を独立に指定できるため、SVG 案を採用しました。

アニメーション同期の要点として、`setInterval(callback, ms)` のタイマーと CSS animation の duration は、両方ともブラウザの requestAnimationFrame ベースで動くので、同じ ms を設定すれば実用上同期します。ただし React の setInterval コールバックからの state 更新は微小な遅延を持ちうるので、CSS animation が終わるタイミングとタイマーが発火するタイミングは厳密に一致しません。この誤差は視覚的にほぼ無視できるレベル（1 フレーム未満）で、ユーザー体験としては「ちょうどリングが空になった瞬間に画像が切り替わる」ように感じられます。

## Decision

**私たちは、スライドショー間隔をユーザーが選択できるようにし、Slideshow トグルボタンを右クリックすることで `5s → 15s → 30s → 60s → 5s` のプリセットを循環させます。選択値は `localStorage` に永続化し、ボタンには現在値のバッジと、再生中は残り時間を示すカウントダウンリングを表示します。**

具体的な設計は次の通りです。

- **プリセット定義**: `client/src/App.tsx` のモジュールスコープに `SLIDESHOW_INTERVALS_MS = [5000, 15000, 30000, 60000] as const` を配置します。モジュール定数として持つことで、再レンダリングのたびに再生成されず、後述の localStorage バリデータからも同じ配列を参照できます。デフォルト値は `DEFAULT_SLIDESHOW_INTERVAL_MS = 5000` です。
- **localStorage キーとバリデーション**: キーは `sumica.slideshow.intervalMs` を使います。読み込み時のバリデータ `loadSlideshowIntervalMs()` は、値が欠損 / 非数値 / プリセット外 / `localStorage` 自体が投げる（Safari プライベートモード等）の場合にデフォルト 5000 にフォールバックします。`useState` の lazy initializer で初回レンダー時のみ実行します。
- **右クリックハンドラ**: Slideshow トグルボタンに `onContextMenu` を追加し、`e.preventDefault()` でブラウザのコンテキストメニューを抑制、`e.stopPropagation()` でライトボックス背景クリックの close ハンドラへの伝播も止めます。`cycleSlideshowInterval()` は現在値のインデックスを `SLIDESHOW_INTERVALS_MS.indexOf(...)` で求め、`(idx + 1) % SLIDESHOW_INTERVALS_MS.length` で次に進みます。プリセット外の値だった場合は 0 番目にリセットします（防御的）。
- **永続化タイミング**: `slideshowIntervalMs` state の変更を `useEffect` で監視し、変わるたびに `localStorage.setItem` を呼びます。`try/catch` で書き込み失敗を握りつぶすことで、書き込み不能な環境でもアプリはクラッシュしません。
- **タイマーとの同期**: [[adr-0040-lightbox-slideshow-and-random-mode-toggle]] のスライドショータイマー `useEffect` の deps 配列に `slideshowIntervalMs` を追加します。これにより間隔切替のたびにタイマーが張り直され、旧間隔の残り時間を待たずに新間隔で再カウントが始まります。
- **tooltip の動的化**: `client/src/i18n/ja.ts` と `en.ts` の `slideshowStartTooltip` を、静的文字列から `(seconds: number) => string` の関数に変更し、Lightbox 側で `t.lightbox.slideshowStartTooltip(Math.round(slideshowIntervalMs / 1000))` として呼び出します。同時に `slideshowCycleIntervalHint` を新設し、tooltip 末尾に「右クリックで間隔を切替」を常時添えて発見性を補います。
- **バッジ表示**: Play/Pause アイコンの右下（`bottom: -2px, right: -2px`）に、`5s` / `15s` / `30s` / `60s` を表示する小さな丸みのある span を配置します。Slideshow の再生 / 停止状態に関係なく常時表示し、現在の設定値を一目で確認できます。
- **カウントダウンリング**: Slideshow が再生中かつ `lightboxIndex >= 0 && displayedHistory.length >= 2` のときのみ、Play/Pause アイコンの上に SVG オーバーレイ（`44 × 44 px`、`viewBox="0 0 44 44"`、`transform: rotate(-90deg)` で 12 時位置から時計回りに開始）を配置します。円は `cx=22, cy=22, r=20`、`stroke-dasharray="125.66"`（= 2π×20 の円周）で、CSS animation `sumica-slideshow-countdown` により `stroke-dashoffset` を `0 → 125.66` に線形補間して 1 周ぶん減らします。`animation-duration` は現在の `slideshowIntervalMs` を inline style で流し込みます。SVG は `key={${lightboxIndex}-${slideshowIntervalMs}}` を持ち、画像が切り替わるたび / 間隔が変わるたびに再マウントされて、animation がリセットされます。
- **prefers-reduced-motion 対応**: `client/src/index.css` の `@media (prefers-reduced-motion: reduce)` で `.sumica-slideshow-countdown-ring` の animation を無効化し、`stroke-dashoffset: 0` の固定表示（完全なリング）にします。アニメーションを避けたいユーザーには残り時間の可視化はしませんが、「スライドショーが動いている」という状態は静的リングで伝えます。

代替案として次を比較検討し、いずれも却下しました。

- **常時見えるプリセットチップ列 (`5s | 15s | 30s | 60s`)**: 発見性は高いが、既に Eye ボタンを退避させたばかりのライトボックスのツールバーをさらに圧迫します。洋一郎さんもミニマル UI を明確に指向されました。
- **クリック展開のドロップダウン**: 開閉 state と outside click / Esc の閉じ判定が増え、`useEffect` が 1 つ以上必要になります。右クリック循環に比べて実装量が大きく、それに見合う UX の利得は薄いと判断しました。
- **`3s / 5s / 8s / 15s / 30s` などの短めプリセット**: 短めに寄せた案でしたが、洋一郎さんは「じっくり見たい」用途を優先して長めのプリセット `5s / 15s / 30s / 60s` を選ばれました。
- **Firestore に永続化**: 端末横断で残るメリットはありますが、この設定は「今この端末で観覧するときの好み」の性質が強く、書き込みコストと依存を増やしてまで同期する動機は弱いと判断しました。`localStorage` で十分です。
- **カウントダウンを 3 桁の秒数テキストで表示**（例: `4.9s → 4.8s → 4.7s`）: 1 秒未満の刻みでテキストが常時更新される UI は読みづらく、視覚ノイズが大きいと判断しました。円周ボーダーの方が「残り時間の割合」を直感で把握でき、可読性が高いです。
- **CSS `conic-gradient` でボタン背景を回転させる**: 実装は SVG より短いですが、色・線幅・開始位置の細かな制御が難しく、既存のボタン背景（`var(--pop-blue)` の塗り）との合成も汚くなりがちでした。SVG オーバーレイの方が独立性が高く、透明背景で重ねられるため採用しました。
- **`requestAnimationFrame` ベースでリングを描画**: React state を毎フレーム更新する形になり、パフォーマンス上のコストが不要に高くなります。CSS animation なら GPU 側で完結し、React の再レンダリングは発生しません。

## Status

承認済み

## Consequences

- **スライドショーの実用範囲が大きく広がりました**。5 秒の「ちらっと見」から 60 秒の「じっくり鑑賞」まで、ユースケースが選べるようになり、特にお気に入り集合を BGM 代わりに流す使い方が現実的になりました。
- **UI 表面積を増やさずに設定を追加できました**。既存の Slideshow トグルボタン内にバッジとリングを重ねるだけで、ツールバーの水平方向を 1 ピクセルも消費していません。同じパターンは将来「他のトグルに設定を持たせる」ときにも参照できる先例になっています。
- **右クリック循環の発見性の弱さは、tooltip の常時ヒント表示（「右クリックで間隔を切替」）で緩和**していますが、tooltip を hover で開かない限りは気づかないという弱点は残っています。実際洋一郎さんが初めて触った際は問題ありませんでしたが、他ユーザーに使わせる場合には気付きにくい可能性があります。ただし本プロジェクトは単独開発であり、実運用上の課題にはなっていません。
- **カウントダウンリングと `setInterval` の同期は視覚的にはほぼ完璧**でした。ブラウザ検証で 2 秒経過時の `strokeDashoffset` が理論値の ±0.03px 以内に収まっており、線形性も確認済みです。将来もし React 18 / 19 の scheduler が変化してタイマー精度が変わっても、CSS animation は独立しているため大きくズレることはないはずです。
- **`localStorage` を初めて明示的に使う機能追加**になりました（[[adr-0031-env-only-config-no-runtime-mutation]] で環境設定の永続化を localStorage から明確に外した経緯とは対照的です）。今回は「ユーザー個人の UI 好み」というスコープに閉じているため localStorage の再導入に問題はありませんが、今後同種の「設定的な値」を UI に持たせる場合はこの機能がロールモデルになります。バリデーション（欠損 / 非数値 / プリセット外）を必ず入れる、というパターンをここで確立しました。
- **i18n キーが関数値になった初のケース**です（従来はすべて静的文字列でした）。TypeScript の型推論は ja / en の shape 一致を保つ形で追随してくれるため、今後 `(x: T) => string` パターンで文字列を作り分けたいときのテンプレートになります。
- **`prefers-reduced-motion` への配慮**が追加されました。ライトボックス周辺のアニメーションはこれまでも [[adr-0025-lightbox-view-transitions]] などで扱ってきましたが、`countdown-ring` のように「本質的に動きが情報を伝える UI」でも、reduce モードでは静的表現に切り替える、という前例が加わりました。今後同種の可視化を追加するときの参考になります。
