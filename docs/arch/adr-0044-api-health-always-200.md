# ADR 44: /api/health は常に HTTP 200 を返し、接続状態はレスポンスボディで表現する

## Context

クライアントは 20 秒ごとに `/api/health` を polling して LM Studio と Stable Diffusion の接続状態を画面右上のバッジに反映します。このエンドポイントの応答設計には次の 2 通りが考えられました。

- (A) LM Studio または SD が接続できないとき HTTP 5xx を返し、body は空 (もしくは最小限のエラーメッセージ) にする
- (B) 常に HTTP 200 を返し、body の `{ lmStudio: { connected: bool, ... }, stableDiffusion: { connected: bool, ... } }` で個別の状態を表現する

クライアント側では、fetch 自体の失敗 (ネットワーク不達、Sumica サーバー自体のダウン) と、アップストリームサービスの一時的な断 (LM Studio や SD が起動していないだけ) を区別してユーザーに提示する必要がありました。

## Decision

私たちは (B) を採用します。`/api/health` は上流サービスの状態にかかわらず HTTP 200 を返し、body の `connected` フラグで LM Studio と SD の個別の可用性を伝えます。

- クライアント側の `fetch` エラーは「Sumica サーバー自体に到達不能」と解釈し、上位のエラー状態にします。
- `fetch` が成功したときは、常に body の `connected` フラグに従って「LM Studio 接続なし」「SD 接続なし」等のバッジを表示します。
- サービスごとのエラー詳細 (タイムアウト、認証失敗等) は body 内の `error` フィールドで補足しますが、クライアントは主にフラグだけを見る単純な実装を保ちます。

(A) を却下した理由は次の通りです。第一に、クライアントで「fetch エラー」と「アップストリーム断」を区別するロジックが必要になり、両者の意味が違うため一律のエラー表示にはできず、条件分岐が増えます。第二に、HTTP 5xx を返すとブラウザ側の DevTools コンソールに赤いエラーが出続け、開発者にとってノイズになります。第三に、Sumica サーバー自身のヘルスチェックとしての意味 (「サーバーは生きている」) と、アップストリーム状態の報告 (「LM Studio は落ちている」) は別次元の情報で、HTTP ステータスコードで両方を表現しようとすると意味が混ざります。

## Status

承認済み

## Consequences

- CLAUDE.md にも「`GET /api/health` pings both upstreams (LM Studio /v1/models, SD /sdapi/v1/sd-models) and always returns 200 with per-service connected flags」と明記され、後続の開発では常に前提となっています。
- クライアント側 (`AppHeader` の接続バッジ) の実装が単純になり、「200 が返ってきたら body を信じる」だけで済みます。
- Sumica サーバーが完全にダウンした場合は fetch そのものが失敗するため、クライアントは「サーバー接続エラー」状態を表示できます。上流サービス断とサーバーダウンが明確に区別されます。
- 副作用として、ヘルスチェックの結果を HTTP ステータスコードで判定するツール (uptime kuma 等の外形監視サービス) を使う場合、「サービスが片方落ちても 200」という挙動が意図と合わない可能性があります。ただし Sumica はローカルツール前提でそのような監視を想定しないため、実害は生じていません。
- 4 秒のタイムアウトを両アップストリームチェックに設定しているため、`/api/health` 応答は最大でも 4 秒程度に収まります。20 秒間隔の polling には十分間に合います。
