# ADR 45: CORS 許可オリジンをホワイトリスト方式にし、接続先 URL は http/https スキームのみを許可する

## Context

初期の Sumica サーバーは `cors()` をデフォルト設定で使い、任意のオリジンからの API 呼び出しを許可していました。加えて、当時存在した `/api/settings` エンドポイントを通じて、ランタイムで LM Studio や Stable Diffusion の接続先 URL を書き換えることが可能でした。

この構成には次のリスクがありました。

- 任意のウェブサイトを開いた状態でユーザーが Sumica のクライアントも起動している場合、そのウェブサイトのスクリプトから Sumica の API を叩けてしまう (CSRF 相当の問題)
- `/api/settings` に不正な URL (`file://`, `javascript:`, `data:`, 内部ネットワークの他ホストなど) を送ることで、SSRF (サーバーサイドリクエストフォージェリ) や DNS rebinding 攻撃の足場になる可能性
- Sumica はローカル開発ツールですが、ユーザーが公開ネットワーク上でクライアントを立ち上げてしまった場合の被害範囲を最小化しておきたい

セキュリティレビューで指摘されたのを機に、CORS ポリシーと接続先 URL のバリデーションを整えることになりました。

## Decision

私たちは次を採用します。

- **CORS**: `CORS_ORIGINS` 環境変数によるホワイトリスト方式にします。default は Vite の開発サーバーで想定される `http://localhost:5173,http://127.0.0.1:5173` の 2 つで、カンマ区切りで複数指定できます。ローカルホストと 127.0.0.1 を両方許容するのは、WSL2 環境などで Windows→WSL のポート転送が localhost しか通らないことがある事情を踏まえたものです。
- **接続先 URL バリデーション** (当時 `/api/settings` に対して): `http:` または `https:` スキームのみ受け付け、それ以外のスキームや不正な URL 形式は 400 エラーで拒否します。ホスト名やポート番号自体の allowlist 化までは行わず、スキームのみを最低限のガードとしました。

CORS のホワイトリスト化については、「default で任意オリジンを許可し続ける」案も検討しましたが、実運用でユーザーがそこにセキュリティ課題を意識しない可能性が高いため、default を狭くしておく方針を優先しました。

## Status

承認済み

## Consequences

- `/api/settings` 自体は後に [[adr-0031-env-only-config-no-runtime-mutation]] で廃止され、接続先設定は `.env` からの読み込みのみに限定されました。URL バリデーションはその廃止と同時に不要になりましたが、CORS の設計は現在も有効で、CLAUDE.md にも「CORS is restricted to the frontend origin(s) ... not wide open」と明記されています。
- 副作用として、Sumica のクライアントを開発者以外のポートやドメイン (例: 別の LAN ホストからブラウザで開く) で配信する場合、`CORS_ORIGINS` を明示的に環境変数で追加する必要があります。運用上の手間はありますが、default を狭くしたトレードオフとして許容範囲です。
- ローカルホストと 127.0.0.1 の両方を許可することで、クライアント側の API ベース URL 計算 (`window.location.hostname` を使う設計) が両ドメインで動作します。これは特に WSL2 環境でハードコードされた `127.0.0.1` が届かないケースに対する暗黙の互換性です。
- SSRF 対策としての URL スキーム検証は `/api/settings` 廃止で不要になったため、現在のコードには残っていません。将来的にサーバー側で外部 URL を扱う機能を追加する場合、この教訓 (スキームは必ず検証する) は再適用の余地があります。
