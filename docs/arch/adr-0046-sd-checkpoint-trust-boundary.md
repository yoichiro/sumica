# ADR 46: SD チェックポイント切替の許可リスト検証は行わず、SD WebUI 自体を信頼境界とする

## Context

`/api/generate` のリクエストボディに含まれる `model` パラメータは、そのまま `override_settings.sd_model_checkpoint` として Stable Diffusion WebUI (AUTOMATIC1111 または Forge) に転送されます。この扱いには次の 2 通りが考えられました。

- (A) サーバー側で許可リスト (現在の `/api/sd-models` から取得可能なチェックポイント名リスト) と照合し、リストにない値は 400 エラーで拒否する
- (B) 検証せず、クライアントが渡した値をそのまま SD WebUI に転送し、SD 側が受け付けるかどうかに委ねる

Sumica は単一ローカルユーザー向けの開発ツールで、SD WebUI もループバックアドレス (通常 `127.0.0.1:7860` または LAN 内) のローカル信頼プロセスとして扱っています。マルチテナントの API サービスではなく、悪意ある任意入力から SD を守る必要のある文脈ではありません。

## Decision

私たちは (B) を採用します。SD チェックポイント名の許可リスト検証を Sumica のサーバー側では実施せず、信頼境界を SD WebUI 自身に置きます。

- クライアントが送ってきた `model` パラメータは、そのまま `override_settings.sd_model_checkpoint` として txt2img ペイロードに含めます。
- 不正な (存在しない) チェックポイント名を渡した場合、SD WebUI 側で「モデルが見つからない」旨のエラー応答が返るため、それをクライアントに転送します。

(A) を却下した主な理由は次の通りです。第一に、SD WebUI 自身が不正なモデル名を安全に扱う (存在しない場合は明示的なエラー応答を返す) 実装になっているため、Sumica 側で重複した検証を挟む必要がありません。第二に、許可リスト検証を追加するにはサーバー側でモデルリストのキャッシュ管理が必要になり、SD 側でモデルが追加されたタイミングとキャッシュ更新の間の一貫性を保つ複雑度が発生します。第三に、Sumica は単一ローカルユーザー向けの信頼された環境を前提としており、任意入力の攻撃面を想定する必要がないため、防御機構としての費用対効果が低いです。

## Status

承認済み

## Consequences

- サーバー側のコードが単純になり、モデルリスト管理を SD 側に完全に委ねられます。SD で新しいチェックポイントを追加してもサーバーの再起動やキャッシュクリアは不要です。
- 副作用として、ユーザーがクライアント経由で誤ったモデル名を送信すると、SD 側からのエラー応答を経てクライアントに戻るまでラウンドトリップが発生します。ただしクライアントは `/api/sd-models` から取得したリストのドロップダウンでしかモデル名を選ばないため、実運用でこの経路が問題になったことはありません。
- 将来的に Sumica をマルチユーザー環境に移行する場合、この判断は再評価対象になります。信頼境界の位置が変わるため、許可リスト検証を含む複数の防御機構を導入する必要が生じます。その際は本 ADR を supersede する新 ADR で議論することになります。
- この考え方は他の SD パススルーパラメータ (`sampler_name`, `scheduler`, `refiner_checkpoint`, `sd_vae` 等) にも共通して適用されており、Sumica 全体の設計として「SD への転送パラメータは基本的にサーバー側で検証しない」という一貫したスタンスを取っています。
