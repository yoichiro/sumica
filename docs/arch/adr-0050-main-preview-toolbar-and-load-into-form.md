# ADR 50: メインプレビュー右列を「常時表示ツールバー＋詳細エリア」構成にし、削除フローを requestDelete で統一する

## Context

元々のメインプレビュー右列は、生成された画像に付随する情報 (拡張プロンプト、モデル、パラメータなど) を表示する静的なパネルでした。ここに次の 2 つの機能を追加する必要が生じました。

- (a) 「フォームにロード」機能: プレビュー中の `currentGeneration` の各値 (プロンプト原文、寸法、Seed、Sampler、Steps、CFG、モデル、LoRA など) を、フォーム state に一括で復元する
- (b) 削除機能: プレビュー中の画像 1 枚を削除する (ギャラリーで既に実装済みの複数選択削除と同じ確認モーダル経由で)

同時期にギャラリー側では既に `POST /api/generations/delete` エンドポイントと確認モーダル ([[adr-0047-delete-flow-client-server-split]]) を持っていたため、プレビュー側の削除フローも同じインフラを使うのが自然でした。しかしプレビュー画面固有の UI 制約 (常に画面下部にツールバーを見せておきたい、詳細情報は縦に長いのでスクロールできる必要がある) もあり、右列全体を再構成する必要が生じました。

## Decision

私たちは次を採用します。

- メインプレビュー右列を **上部: 常時表示ツールバー** (「フォームにロード」ボタン、「削除」ボタン) と **下部: スクロール可能な詳細エリア** の 2 段構成に再構成します。
- 「フォームにロード」ボタンを押すと、`currentGeneration` の各値をフォーム state に復元します。**Seed は復元と同時に「Seed をロックする」チェックボックスも自動的に ON** にして、再現性を担保します。
- 削除フローは `requestDelete(ids: string[])` という汎用関数として抽出し、ギャラリーの複数選択削除とプレビューの単体削除の両方から呼び出します。`DeleteConfirmModal` の表示から、[[adr-0047-delete-flow-client-server-split]] の 2 経路 (Firebase SDK 経由またはローカルエンドポイント経由) の実行までを、この関数が一元的に扱います。
- 新規エンドポイントは追加しません。既存の `POST /api/generations/delete` (ローカルモード) と Firebase SDK 呼び出し (サインイン時) をそのまま流用します。

代替案として、「フォームにロード」を新規エンドポイント経由 (サーバーで復元パラメータを組み立てる) にする案も検討しましたが、フォーム state の管理は既にクライアント側で完結しているため、サーバーを介する意味がありませんでした。同様に、削除確認モーダルをプレビュー専用に新設する案も検討しましたが、UX の一貫性のためにギャラリーと同じモーダルを共用しました。

## Status

承認済み

## Consequences

- 「フォームにロード」は Sumica の中核機能として定着し、テストにも `client/src/components/loadIntoFormState.test.ts` として明示的に単体テストが追加されました。CLAUDE.md でも中核機能として言及されています。
- [[adr-0024-ranking-recipe-full-form-restore]] (ランキングレシピをフォーム全復元可能な次元まで拡張) は、この「フォームにロード」の基盤の上に構築されました。ランキング画面から「レシピを適用」する UX も、内部的にはこの復元機構と同じ `loadIntoFormState` を経由します。
- 削除フローの重複が排除されたことで、後続の機能追加がスムーズになりました。[[adr-0019-gallery-shift-click-range-selection]] の Shift+クリック範囲選択、[[adr-0034-gallery-d-key-delete-shortcut]] の D キーショートカット、[[adr-0033-lightbox-eye-button-open-in-preview]] からのプレビュー削除は、いずれも同じ `requestDelete` を呼び出す形で追加され、削除ロジックの散在を防いでいます。
- Seed 自動ロック ON は最初は驚きを与える可能性があった (「フォームにロードすると勝手にロックがかかる」) ものの、実運用では「同じ画像を再生成できる」という期待に応える挙動として好評でした。ユーザーが再現性を望まない場合は、明示的にロックを外す必要がある UI コストは残ります。
- 常時表示ツールバー + スクロール可能な詳細エリアの 2 段構成は、CSS Grid で `gridTemplateRows: 'auto minmax(0, 1fr)'` として実装され、後の [[adr-0042-flux-support-3way-architecture]] の Flux 対応でも同じレイアウトパターンが踏襲されています。
