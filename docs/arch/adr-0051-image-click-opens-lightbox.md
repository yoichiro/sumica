# ADR 51: 画像クリック挙動を「クリック=ライトボックス起動」に統一し、選択操作を専用アイコンに分離する

## Context

[[adr-0048-lightbox-overlay-foundation]] でライトボックスを導入した当初、履歴ギャラリーのタイルは次の相互作用モデルでした。

- タイルの画像本体をクリック: 選択/選択解除 (青いチェックバッジが付く)
- タイル右下の虫めがねアイコンをクリック: ライトボックスで拡大表示

このモデルには次の課題がありました。第一に、画像を拡大表示したい (直感的な期待) のに、ユーザーはまず右下の小さなアイコンを狙う必要があり、モバイル/タッチデバイスでは操作性が悪くなります。第二に、ライトボックス側では [[adr-0027-lightbox-space-selection-toggle]] で Space キーによる選択トグルを追加したため、選択操作が「クリック」と「Space キー (ライトボックス内)」の 2 系統に分裂し、一貫性が欠けていました。第三に、「クリック = 選択」の挙動は Windows Explorer 風で PC 慣れしたユーザーには馴染むものの、Gmail や Google Photos のような近年のウェブアプリでは「クリック = 開く」が主流になっており、Sumica の UI の期待値もそちらに寄っていました。

洋一郎さんから、画像クリックを「拡大表示」の挙動にしたい、選択操作は独立して欲しい、という要望がありました。次の 2 通りを比較しました。

- (A) クリックで選択、ダブルクリックで拡大表示 (Windows Explorer 風)
- (B) クリックで拡大表示、選択操作は専用アイコン (Gmail/Google Photos 風)

## Decision

私たちは (B) を採用します。

- 画像本体のクリック → **ライトボックスを開く** (`onOpenLightbox` を呼び出す)
- 選択操作 → タイルに配置する **専用の選択アイコンボタン** (ライトボックスの選択ボタン [[adr-0027-lightbox-space-selection-toggle]] とデザインを統一)
- カーソルは `pointer` にして、画像がクリック可能な要素であることを視覚化します。

(A) を却下した主な理由は次の通りです。第一に、ダブルクリックは学習コストが高く、モバイル環境でのタップ 2 回は誤操作やタッチイベントの遅延が発生しやすい操作です。第二に、ダブルクリック待機のための遅延 (通常 300ms 程度) が単クリックのレスポンスを鈍らせます。第三に、Gmail や Google Photos などのモダンなウェブアプリの操作感覚と揃うことで、ユーザーの直感的な期待に応えられます。

## Status

承認済み

## Consequences

- 画像 = 拡大するもの、専用アイコン = 選択するもの、という役割分担が明確になり、直感的な操作性を実現しました。
- 後続の関連機能はこのモデルの上に自然に載りました。[[adr-0019-gallery-shift-click-range-selection]] の Shift+クリック範囲選択は選択アイコンに対して動作します。[[adr-0033-lightbox-eye-button-open-in-preview]] の Eye ボタンはメインプレビューへの反映として独立した第 3 の操作を提供します。[[adr-0034-gallery-d-key-delete-shortcut]] の D キーはキーボード派向けの補完です。
- Sumica のコード上、`HistoryGallery.tsx` の画像は `onClick={onOpenLightbox}` に、選択アイコンは別要素の `onClick={handleSelectClick}` に分離される形で実装されています。関数名の分離自体がこの ADR の判断を反映しています。
- モバイル対応は依然として未整備のままです。Sumica の想定利用環境が SD 実行機と同居するデスクトップ主体であることから、モバイル UI の優先度は低く保たれています。将来的にモバイル対応を検討する場合、タッチイベントとホバーイベントの扱いを含めて再度検討することになります。
- ダブルクリックを予約していないことで、将来的に別の操作 (例: 画像をコピー、外部ビューアで開くなど) に割り当てられる余地は残っています。ただし現時点でそういった要求はなく、YAGNI として保留しています。
