# UI 国際化（i18n）設計

## 背景と目的

Sumica AI Studio の UI は現在すべて日本語で書かれています。約 176 個の日本語文字列が 13 個のクライアントファイル（`App.tsx` および 12 個のコンポーネント / ユーティリティモジュール）に散在しており、日本語話者以外は使えない状態です。

洋一郎さんから、UI を英語にも対応させたいというリクエストがありました。要件は次の通りです。

- 表示言語は日本語（`ja`）と英語（`en`）の 2 種類をサポートする。
- **locale 決定の優先順位**は URL クエリパラメータ `?hl=ja` / `?hl=en` → ブラウザの `navigator.language` が `ja` で始まれば `ja` → それ以外は `en` にフォールバック。
- アプリ内に**言語切替 UI（ドロップダウンやトグル）は置かない**。読み込み時に一度だけ決定する。
- サーバーサイド（`server/*.ts`）のメッセージは触らない。既に英語で返されているエラー詳細をクライアント側の i18n 辞書で日本語 / 英語のラップメッセージに埋め込む形にする。

## 決定

**外部 i18n ライブラリを採用せず、カスタム軽量モジュール（`client/src/i18n/`）を実装します。**

具体的な構成は次の通りです。

- `client/src/i18n/index.ts` : `resolveLocale()` 関数と `locale` / `t` の named export。
- `client/src/i18n/ja.ts` : 日本語文字列を機能領域ごとにネストしたオブジェクトとして export。
- `client/src/i18n/en.ts` : 英語文字列。`typeof ja` を型注釈にして構造整合を TypeScript コンパイル時に強制。
- `client/src/i18n/index.test.ts` : `resolveLocale()` の分岐を Vitest で検証。

代替案として次を比較検討し、いずれも却下しました。

- **react-i18next / react-intl / FormatJS を採用**: 実績のあるライブラリで機能豊富ですが、Sumica はローカル・シングルユーザー向けの小規模ツールでプルーラル形式、日付ローカライズ、通貨などの複雑な i18n 機能を要さず、ライブラリの `50〜100KB` の追加バンドルは重すぎる。学習コストもかかる。
- **ヘッダーに言語切替 UI を追加**: 便利ではあるが優先順位のロジックが増え、localStorage による永続化と URL / ブラウザ設定との干渉ルールを決める必要が出る。今回の要件では load-time 決定で十分と判断。
- **サーバーサイド i18n（`Accept-Language` ヘッダ経由）**: サーバーコードを触ることになり範囲が広がる。現在のサーバーエラー文字列は短く、ユーザーが直接目にする機会も少ないため、クライアント側のラップメッセージで包む方針で十分。
- **文字列を機能領域ごとに別ファイルに分割**（例 `i18n/header.ts`, `i18n/toast.ts`）: 176 文字列は 1 ファイルに収まるサイズで、複数ファイルにすると import が増え、キーの探しやすさが下がる。1 ファイル per locale で組織するのが素直。

## アーキテクチャ

### locale 決定ロジック

`client/src/i18n/index.ts` の中核実装は次の形になります。

```ts
import { ja } from './ja';
import { en } from './en';

export type Locale = 'ja' | 'en';

export function resolveLocale(): Locale {
  const hl = new URLSearchParams(window.location.search).get('hl');
  if (hl === 'ja') return 'ja';
  if (hl === 'en') return 'en';
  return navigator.language.toLowerCase().startsWith('ja') ? 'ja' : 'en';
}

export const locale: Locale = resolveLocale();
export const t = locale === 'ja' ? ja : en;

if (typeof document !== 'undefined') {
  document.documentElement.lang = locale;
}
```

決定の優先順位は次の順で評価します。

1. **URL クエリ `?hl=ja`** → `'ja'`
2. **URL クエリ `?hl=en`** → `'en'`
3. **URL クエリなし、または `hl` が `ja` / `en` 以外**（`?hl=fr` など）→ ブラウザ言語判定に進む
4. **`navigator.language`** が `ja`（`ja`, `ja-JP` 等）で始まる → `'ja'`
5. **それ以外**（`en-US`, `fr-FR` 等）→ `'en'`（フォールバック）

`hl` の値が想定外の場合はエラーを投げず、静かに 3〜5 のフォールバック経路に落とします。ユーザーが URL を手入力してタイポしても壊れないため。

### モジュール実行タイミング

- `locale` と `t` は**モジュールロード時に一度だけ評価**される（`export const`）。
- 実行時の切替は不可（要件外）。
- Vite の CSR 前提で、SSR / prerender は非対応。

### `<html lang="...">` の副作用

`document.documentElement.lang` を実際の locale に設定します。この副作用は `index.ts` の import 時点で発火するので、副作用を持つモジュール依存関係の一部として `main.tsx` から自然に走ります。`index.html` の初期値 `<html lang="ja">` は残しますが、JS 実行直後に上書きされるので、実質的な言語は `resolveLocale()` の結果になります。

### 文字列辞書の型設計

`ja.ts` は次の形のオブジェクトを default export しません（named export の `ja` を持ちます）。機能領域ごとにネストします。

```ts
export const ja = {
  header: {
    title: 'Sumica AI Studio 🎨⚡️',
    subtitle: 'Creative Image Lab',
    signInFailed: (msg: string) => `サインインに失敗しました: ${msg}`,
    // ...
  },
  controlPanel: { /* ... */ },
  preview: { /* ... */ },
  gallery: { /* ... */ },
  lightbox: { /* ... */ },
  batchModal: { /* ... */ },
  deleteConfirm: { /* ... */ },
  toast: {
    generateSuccess: '画像を生成しました！🎨⚡️',
    generateFailed: (details: string) => `画像生成に失敗しました。\n\n詳細: ${details}\n\nLM Studio や Stable Diffusion がローカルで正常に起動しているか確認してください。`,
    batchAllSuccess: (n: number) => `${n}枚の画像を生成しました！🎨⚡️`,
    batchPartial: (total: number, ok: number, fail: number) =>
      `${total}枚中${ok}枚を生成しました（${fail}枚失敗）。\n\nLM Studio や Stable Diffusion がローカルで正常に起動しているか確認してください。`,
    // ...
  },
  notification: {
    generateSuccess: '画像生成が完了しました 🎨',
    // ...
  },
};
```

**補間の統一形式**: 補間が要る文字列は**関数**、要らないものは**プレーン文字列**として定義します。呼び出し側は `t.toast.generateSuccess`（文字列）または `t.toast.batchAllSuccess(10)`（関数呼び出し）と自然に使い分けます。

`en.ts` は同じキー構造を持つオブジェクトを、次のように型注釈付きで export します。

```ts
import type { ja } from './ja';

export const en: typeof ja = {
  header: {
    title: 'Sumica AI Studio 🎨⚡️',
    subtitle: 'Creative Image Lab',
    signInFailed: (msg) => `Sign-in failed: ${msg}`,
    // ...
  },
  // ...
};
```

`typeof ja` を型として使うことで、`en.ts` に**キーが 1 つでも欠けている / 余分にある** / **関数のシグネチャが違う**場合は TypeScript コンパイルエラーになります。これが翻訳漏れ検出の主な仕組みで、Vitest テストは不要です。

### サーバーメッセージのラップ

現状のサーバー（`server/index.ts`）は次のような英語エラーを返します。

```ts
res.status(400).json({ error: 'Prompt is required' });
res.status(500).json({ error: 'Image generation pipeline failed.' });
```

クライアントはこれをそのまま UI に表示せず、i18n 辞書の**ラップメッセージ内の details 引数**として埋め込みます。

```ts
// 現状（変更前）
addToast(`画像生成に失敗しました。\n\n詳細: ${errData.error}`, 'error');

// 変更後
addToast(t.toast.generateFailed(errData.error), 'error');
```

日本語 UI: `画像生成に失敗しました。詳細: Prompt is required`
英語 UI: `Image generation failed. Details: Prompt is required`

サーバー側のコードには 1 行も変更を加えません。

## 移行の順序

コミット単位を細かく分けて安全に進めます。

1. **インフラ構築**: `i18n/index.ts` + `resolveLocale()` の Vitest + 空の（あるいは最小限の）`ja.ts` / `en.ts` を作成。既存コードは無変更。この段階では build も動作も現状のまま。
2. **文字列抽出（完全版 `ja.ts`）**: 13 ファイルから全 UI 文字列を `ja.ts` に転記し、機能領域ごとにネスト構造で整理。呼び出し側はまだ変えないので、この時点でアプリの見た目は何も変わらない。
3. **英訳（完全版 `en.ts`）**: `typeof ja` を型注釈にして構造整合を強制。`ja.ts` の内容を全て英語化。tsc 通過で構造整合を保証。
4. **各コンポーネント / モジュールを i18n 化**: 以下のファイルを 1 つずつ変換していく。1 コミット = 1 ファイル（あるいは論理的に近い数個）を目安に。
   - `components/AppHeader.tsx`（最初にパターン確立）
   - `components/ControlPanel.tsx`
   - `components/PreviewPanel.tsx`
   - `components/HistoryGallery.tsx`
   - `components/Lightbox.tsx`
   - `components/BatchGenerationModal.tsx`
   - `components/DeleteConfirmModal.tsx`
   - `components/ToastContainer.tsx`
   - `components/GenerationBadge.tsx`
   - `components/captionFields.ts`（`'不明'` のフォールバックのみ）
   - `components/loadIntoFormState.ts`（UI 文字列があれば）
   - `utils/notifications.ts`（通知本文のみ、タイトル `Sumica AI Studio` は固定）
   - `App.tsx`（全トースト呼び出しと残りの UI 文字列）

各コンポーネントの変換はほぼ機械的な作業です：日本語文字列を見つけて `t.section.key` に置き換え、必要に応じて `import { t } from '../i18n'` を追加します。

## テスト戦略

### Vitest による自動テスト

`client/src/i18n/index.test.ts` で `resolveLocale()` の分岐を検証します。

- URL クエリ `?hl=ja` → `'ja'`
- URL クエリ `?hl=en` → `'en'`
- URL クエリ `?hl=fr`（無効値）→ ブラウザ言語フォールバック
- URL クエリなし + `navigator.language = 'ja-JP'` → `'ja'`
- URL クエリなし + `navigator.language = 'ja'` → `'ja'`
- URL クエリなし + `navigator.language = 'en-US'` → `'en'`
- URL クエリなし + `navigator.language = 'fr-FR'` → `'en'`（フォールバック）

Vitest では `window.location` と `navigator.language` を `beforeEach` でモックします。jsdom のデフォルトが `'en-US'` である前提で、必要な case だけ上書き。

### TypeScript による構造保証

`ja.ts` と `en.ts` のキー構造整合は `typeof ja` の型注釈で強制されるため、翻訳漏れは `tsc` のコンパイル時に検出されます。専用の Vitest テストは書きません。

### 手動確認

以下の 6 パターンを手動でブラウザ確認します。

- `?hl=ja` でリロード → 日本語表示
- `?hl=en` でリロード → 英語表示
- `?hl=fr` でリロード → ブラウザ言語に応じてフォールバック
- ブラウザ言語 = JP でクエリなし → 日本語
- ブラウザ言語 = EN でクエリなし → 英語
- 各画面（プロンプト入力、詳細設定パネル、プレビュー進捗、ギャラリー、ライトボックス、バッチモーダル、削除確認モーダル、各種トースト、OS 通知）をひと通り英語版で見てレイアウトが崩れていないか確認

## 移行しないもの

以下は日本語のまま（あるいは既存の言語のまま）残します。

- **`console.log` / `console.error` 内のメッセージ**: 開発者向けデバッグ、UI ではない。
- **コード中のコメント**: 開発者向け、UI ではない。
- **`client/index.html` の `<title>` 属性** `Sumica AI - Creative Image Studio`: 既に英語なので変更不要。
- **サーバーサイド `server/*.ts`**: 完全にノータッチ。エラーメッセージは英語のまま返され、クライアントの i18n 辞書がラップする。
- **ADR / 設計仕様書 / 実装計画などのドキュメント（`docs/`）**: 日本語のまま。
- **CLAUDE.md**: 日本語のまま。
- **`SDXL_PRESETS` のラベル文字列 `'S'`, `'M'`, `'L'`, `'1:1'`, `'landscape'` 等の記号的な定数**: UI に露出するがローカライズ不要な記号として扱う。

## レイアウト崩れの対策

英語化するとテキストが長くなりがちなので、次のポイントは特に手動確認で見ます。

- **ヘッダーステータスバー**: `LM Studio 接続中` (10 文字) → `LM Studio connected` (17 文字)。既存の `text-overflow: ellipsis` と最大幅制限で吸収できるはず。
- **バッチモーダルの segmented tab**: `画像枚数`, `サイズ`, `モデル` → `Count`, `Sizes`, `Models`（英語のほうが短くて幸い）。
- **ボタンラベル**: `生成する` (4 文字) → `Generate` (8 文字)。ボタンの min-width を確認。
- **ギャラリーのキャプション行**: 現状は狭い領域だが、モデル名は同じ、サイズは同じ、日付フォーマットも同じなので影響なし。
- **プレビュータブラベル**: `🎨 プレビュー＆進捗`, `🖼️ 履歴ギャラリー (N)` の絵文字と件数の位置関係が英語版でも崩れないか確認。

必要に応じて英語版だけ font-size やパディングを調整することも可能ですが、まずは共通スタイルで様子を見ます。

## 影響範囲

- **変更ファイル**: `client/src/App.tsx`, `client/src/components/*.tsx`（12 ファイル）, `client/src/utils/notifications.ts`
- **新規ファイル**: `client/src/i18n/index.ts`, `client/src/i18n/ja.ts`, `client/src/i18n/en.ts`, `client/src/i18n/index.test.ts`
- **サーバー**: 無変更。
- **Firebase / Firestore / Storage スキーマ**: 無変更。
- **依存パッケージ**: 無変更（外部ライブラリ不採用）。
- **バンドルサイズ**: `ja.ts` と `en.ts` の両方が含まれるので数 KB 増加するが、外部ライブラリを入れるより遥かに軽量。将来 tree-shake で未使用 locale を除外したくなれば、dynamic import による分割ロードに移行可能。

## ADR 化

今回の i18n 導入はプロジェクトの i18n 戦略を規定するアーキテクチャ上の意思決定です。実装完了後に ADR（`adr-0020-` 付近）を作成します。ADR で記録する内容：

- なぜカスタム軽量モジュールを採用し、react-i18next / react-intl / FormatJS を却下したのか
- locale 決定の優先順位（URL param → browser language → EN fallback）
- サーバーメッセージは英語のまま、クライアントで詳細をラップする責務分担
- 言語切替 UI を今回は追加しない判断と、将来必要になった場合の拡張余地
